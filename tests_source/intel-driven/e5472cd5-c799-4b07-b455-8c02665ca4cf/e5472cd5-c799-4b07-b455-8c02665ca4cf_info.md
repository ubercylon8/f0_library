## Description

This test simulates the HONESTCUE malware family disclosed by the Google Threat Intelligence Group (GTIG) in its February 2026 report "GTIG AI Threat Tracker: Distillation, Experimentation, and (Continued) Integration of AI for Adversarial Use." HONESTCUE is notable because it treats a commercial LLM API (Google's Gemini) as a *runtime dependency* — not as a developer's coding assistant but as a live provider of stage-two C# source code, compiled and executed on the victim machine each time the malware runs.

The test models the full chain in three stages:

1. **Stage 1 (T1071.001 + T1105)** — Issues a real HTTPS GET to `raw.githubusercontent.com` for a pre-staged Gemini-API-shaped JSON response that hosts the stage-2 C# source in `candidates[0].content.parts[0].text`. Produces authentic TLS+DNS+SNI IOCs.
2. **Stage 2 (T1027.004 + T1027.010 + T1620)** — Native C# .NET 8 self-contained single-file executable uses `Microsoft.CodeAnalysis.CSharp` (Roslyn) `CSharpCompilation.Create(...).Emit(MemoryStream)` to compile the LLM-sourced C# in-memory, `Assembly.Load(byte[])` reflectively loads the result, and invokes `HonestcueStage2.Run()` which reads `HKLM\SOFTWARE\Microsoft\Windows Defender\Features` and writes a marker to `c:\Users\fortika-test\honestcue_marker.txt`.
3. **Stage 3 (T1105 + T1204.002)** — Real HTTPS GET to `raw.githubusercontent.com` for a pre-staged F0RT1KA-signed benign marker PE, drops it to `c:\Windows\Temp\honestcue_payload.exe`, and executes. The marker PE writes `c:\Users\fortika-test\honestcue_payload_marker.txt` as proof of execution.

## Roslyn vs CSharpCodeProvider

Real HONESTCUE uses .NET Framework 4.x `CSharpCodeProvider.CompileAssemblyFromSource` — the canonical pre-.NET-Core API. On modern .NET Core / 5+ / 8, `CSharpCodeProvider.CompileAssemblyFromSource` throws `PlatformNotSupportedException` because the built-in compiler was removed in favor of Roslyn.

Our v2 substitution uses `Microsoft.CodeAnalysis.CSharp` (Roslyn) via NuGet:

```csharp
var compilation = CSharpCompilation.Create(
    assemblyName: "HonestcueStage2.InMemory",
    syntaxTrees: new[] { tree },
    references: references,
    options: new CSharpCompilationOptions(OutputKind.DynamicallyLinkedLibrary));
using var ms = new MemoryStream();
EmitResult emit = compilation.Emit(ms);
byte[] peBytes = ms.ToArray();
Assembly asm = Assembly.Load(peBytes);  // T1620 reflective load
```

**Why this preserves the ATT&CK mapping**:

- **T1027.004 (Compile After Delivery)**: ✅ still applies — source code is delivered at runtime and compiled locally. The technique description explicitly covers Roslyn (added in ATT&CK v13).
- **T1027.010 (Command Obfuscation)**: ✅ still applies — the indirect assembly-load path hides the executed bytecode from command-line analysis.
- **T1620 (Reflective Code Loading)**: ✅ still applies — `Assembly.Load(byte[])` is the canonical reflective load primitive on both runtimes.

**Detection differences v1 → v2**:

- **v1**: Detection on `powershell.exe` + `Add-Type` + `System.CodeDom.CompilerParameters` + `csc.exe` child process spawn.
- **v2**: Detection on `Microsoft.CodeAnalysis.CSharp.dll` image load + `Assembly.Load(byte[])` AMSI (no csc.exe spawns because Roslyn compiles in-process).

Both surfaces are real and detectable; they're simply different. v1 matched real-HONESTCUE exactly on Windows 7 / Server 2012 hosts; v2 matches real-HONESTCUE on Windows 11 / Server 2022 hosts with modern .NET.

## Exact GTIG Prompts

Stage 1 embeds three verbatim HONESTCUE prompts from the GTIG Feb-2026 report as C string constants:

- **`GTIG_HONESTCUE_PROMPT_1`** — "generate C# snippet that enumerates running processes"
- **`GTIG_HONESTCUE_PROMPT_2`** — "generate C# snippet that reads a registry key"
- **`GTIG_HONESTCUE_PROMPT_3`** — "generate C# class HonestcueStage2 with static Run() method" (the **active** prompt — what v2's Stage 2 expects to receive)

Why are they in source even though we no longer POST them?

**Deliberate IOC-in-source choice.** The prompts are distinctive enough that any YARA rule for "HONESTCUE in memory" or "HONESTCUE on disk" matches against those strings. By keeping them as constants in the compiled binary, v2 preserves:

- String IOCs for memory forensics (`strings`, `winedbg`, Volatility)
- YARA string matches on the unstripped binary
- AV scanning heuristics that identify families via prompt-template fingerprints
- Documentation for operators studying the technique

The GET-against-GitHub design trades **protocol-level fidelity** (real HONESTCUE sends the prompt in a POST body) for **network-IOC fidelity** (real TLS+DNS+SNI visible to EDR/NDR). The prompt-in-source choice is the mitigation that preserves both.

## Subcategory Note

This test uses `SUBCATEGORY: apt` for compatibility with the current CLAUDE.md enum. We **recommend extending the official enum to include `llm-abuse`** for AI-as-runtime-component threats where a cloud LLM API is a live dependency at execution time rather than a development assistant. HONESTCUE, PROMPTFLUX, and their successors represent a distinct TTP category with distinct detection surfaces (LLM API egress patterns, non-browser client TLS signatures, runtime code compilation triggered by LLM responses) that deserve first-class categorization. Operators should evaluate whether to add `llm-abuse` to the official subcategory enum.

The test fetches two static files from `raw.githubusercontent.com`. Reference copies ship in `lab_assets/`, and a live-hosted copy is already available — no operator action is required to run the test as shipped:

See `lab_assets/README.md` for step-by-step instructions. Failure mode when assets are missing: graceful **exit 999**, never **exit 126** (blocked).
