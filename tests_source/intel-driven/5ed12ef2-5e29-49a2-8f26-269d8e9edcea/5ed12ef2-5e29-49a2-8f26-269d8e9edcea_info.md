## Description

This test simulates a complete modern ransomware attack killchain from initial execution through final impact. Unlike single-technique tests, this multi-stage architecture executes 5 distinct ATT&CK techniques sequentially, stopping at the first blocked stage to provide technique-level detection precision.

The test evaluates EDR/AV effectiveness against realistic ransomware behaviors including privilege escalation, aggressive file discovery, rapid encryption simulation, and ransom note deployment. Each stage is implemented as a separate signed binary, allowing precise identification of where defensive controls activated.

This simulation is based on documented ransomware behaviors from threat actors including Conti, LockBit, and BlackCat/ALPHV, providing high-fidelity testing against real-world attack patterns.

## Technical Details

### Multi-Stage Architecture

This test uses F0RT1KA's multi-stage pattern where each ATT&CK technique is implemented as a separate signed binary:

**Main Orchestrator**: `5ed12ef2-5e29-49a2-8f26-269d8e9edcea.exe`
- Extracts 5 signed stage binaries to C:\F0
- Executes stages sequentially
- Monitors exit codes after each stage
- Stops immediately if any stage is blocked
- Evaluates final result based on stage completion

**Stage Binaries**:
1. `5ed12ef2-5e29-49a2-8f26-269d8e9edcea-T1204.002.exe` - Initial Execution
2. `5ed12ef2-5e29-49a2-8f26-269d8e9edcea-T1134.001.exe` - Privilege Escalation
3. `5ed12ef2-5e29-49a2-8f26-269d8e9edcea-T1083.exe` - File Discovery
4. `5ed12ef2-5e29-49a2-8f26-269d8e9edcea-T1486.exe` - Encryption
5. `5ed12ef2-5e29-49a2-8f26-269d8e9edcea-T1491.001.exe` - Ransom Note Deployment

Each stage binary is signed BEFORE embedding in the main orchestrator, ensuring Windows trusts the binaries when extracted.

### Attack Flow

#### Stage 1: User Execution - Malicious File (T1204.002)

**Objective**: Simulate initial ransomware payload execution

**Actions**:
- Drops multiple ransomware component files to C:\F0:
  - `crypto_engine.dat` - Encryption engine marker
  - `file_enum.dat` - File discovery module marker
  - `ransom_core.dat` - Core ransomware logic marker
- Tests ability to create and execute batch scripts
- Verifies process spawn capability
- Creates stage completion marker

**Detection Indicators**:
- Rapid creation of multiple .dat files
- Suspicious file names (crypto_engine, ransom_core)
- Process spawning behavior
- Batch script execution

**EDR Should Detect**: Suspicious file drops to non-standard directory, rapid multi-file creation pattern

#### Stage 2: Access Token Manipulation (T1134.001) - CRITICAL

**Objective**: Attempt privilege escalation to SYSTEM

**Actions**:
- Opens current process token with `OpenProcessToken`
- Checks token elevation status
- Enumerates processes to find SYSTEM processes (winlogon.exe, lsass.exe)
- Attempts `OpenProcess` with PROCESS_QUERY_INFORMATION on SYSTEM processes
- Tries to duplicate SYSTEM token
- Attempts to enable dangerous privileges:
  - `SeDebugPrivilege` - Process debugging capability
  - `SeBackupPrivilege` - Bypass file permissions
  - `SeRestorePrivilege` - Bypass file permissions
  - `SeTakeOwnershipPrivilege` - Take ownership of files
  - `SeShutdownPrivilege` - System shutdown capability

**Detection Indicators**:
- OpenProcessToken API calls
- Process enumeration (CreateToolhelp32Snapshot)
- Access attempts to protected SYSTEM processes
- AdjustTokenPrivileges API calls for dangerous privileges
- Token duplication attempts

**EDR Should Detect**: Token manipulation attempts, access to SYSTEM processes, privilege escalation behavior

**Most Likely Blocking Point** on protected systems.

#### Stage 3: File and Directory Discovery (T1083)

**Objective**: Enumerate file system for high-value encryption targets

**Actions**:
- Creates test document directory: `C:\F0\test_documents`
- Populates with various file types (docx, xlsx, pdf, db, jpg, zip, key files)
- Scans multiple directories:
  - C:\F0 (test directory)
  - User's Desktop
  - User's Documents
  - User's Downloads
- Identifies files by extension priority:
  - **Priority 1 (Critical)**: .docx, .xlsx, .pdf, .db, .pem, .key
  - **Priority 2 (Important)**: .cpp, .java, .py, .zip
  - **Priority 3 (Standard)**: .jpg, .mp3, .mp4
- Creates target list file: `C:\F0\targets.list`
- Simulates shadow copy and network share discovery (logged only)

**Detection Indicators**:
- Recursive directory traversal across multiple user directories
- File extension filtering behavior
- Creation of target list file
- Rapid file system enumeration (100+ files)

**EDR Should Detect**: Aggressive file enumeration pattern, target list creation

#### Stage 4: Data Encrypted for Impact (T1486) - CRITICAL

**Objective**: Simulate rapid file encryption (primary detection point)

**Actions**:
- Generates AES-256 encryption key
- Creates AES-GCM cipher for authenticated encryption
- Enumerates files in test directory
- **Rapid file encryption**:
  - Reads original file
  - Encrypts with AES-256-GCM
  - Writes encrypted file with `.f0rtika` extension
  - Deletes original file
  - Processes multiple files per second (aggressive rate)
- Creates `MASTER_KEY.key` file (simulated key escrow)
- Logs (but does NOT execute) destructive commands:
  - `vssadmin delete shadows /all /quiet` - Delete volume shadow copies
  - `bcdedit /set {default} recoveryenabled no` - Disable recovery
- Creates encryption summary file

**Detection Indicators**:
- Rapid file read/write operations (10+ files/second)
- File extension changes (original → .f0rtika)
- Original file deletions
- High entropy file writes (encrypted data pattern)
- Mass file operations in short time window
- MASTER_KEY.key creation
- Encryption summary file

**EDR Should Detect**: Mass file modification pattern, rapid encryption behavior, suspicious file extension changes

**Second Most Likely Blocking Point** - behavioral ransomware detection should trigger here.

#### Stage 5: Defacement - Internal Defacement (T1491.001)

**Objective**: Deploy ransom notes and deface system

**Actions**:
- Creates text ransom note (`README_RANSOMWARE.txt`)
- Creates HTML ransom note (`YOUR_FILES_ENCRYPTED.html`)
- Deploys notes to multiple locations:
  - C:\F0
  - User Desktop
  - User Documents
  - User Profile directory
  - Public Desktop
- Attempts desktop wallpaper modification via registry
- Attempts to auto-open ransom note in Notepad
- Creates payment instructions file (simulated)

**Detection Indicators**:
- Multiple identical files written to different locations
- Desktop and Documents folder modifications
- Registry writes to wallpaper settings
- Notepad.exe spawned with suspicious file argument
- HTML content with ransomware messaging

**EDR Should Detect**: Multiple ransom note deployments, registry modification attempts, suspicious file operations

## Behavioral Detection

For organizations wanting to develop detection rules based on this test, consider the following KQL queries for Microsoft Sentinel/Defender:

### Mass File Operations Detection
```kql
DeviceFileEvents
| where TimeGenerated > ago(1h)
| where ActionType in ("FileModified", "FileDeleted", "FileCreated")
| where FolderPath !startswith "C:\\F0\\"  // Exclude test directory
| summarize FileCount=count(),
    UniqueExtensions=dcount(FileExtension),
    Operations=make_set(ActionType)
    by DeviceName, InitiatingProcessFileName, AccountName, bin(TimeGenerated, 5m)
| where FileCount > 50  // More than 50 files in 5 minutes
| extend ThreatScore = case(
    FileCount > 100, 90,
    FileCount > 75, 75,
    60
)
```

### Token Manipulation Detection
```kql
DeviceProcessEvents
| where TimeGenerated > ago(1h)
| where ProcessCommandLine has_any ("winlogon", "lsass")
    or InitiatingProcessFileName =~ "powershell.exe"
| where ActionType == "ProcessAccess"
| summarize AccessCount=count(),
    TargetProcesses=make_set(FileName)
    by DeviceName, InitiatingProcessFileName, AccountName
| where AccessCount > 3
```

### File Extension Changes Detection
```kql
DeviceFileEvents
| where TimeGenerated > ago(1h)
| where ActionType == "FileRenamed"
| where FileName has_any (".f0rtika", ".encrypted", ".locked")
| summarize RenamedFiles=count()
    by DeviceName, InitiatingProcessFileName, AccountName, bin(TimeGenerated, 5m)
| where RenamedFiles > 5
```

## Safety and Recovery

### Safety Mechanisms

1. **Isolated Test Environment**:
   - All operations in C:\F0 directory
   - Test creates its own files (no user data at risk)
   - Clearly labeled ransom notes (security test markers)

2. **Simulated Destructive Commands**:
   - Shadow copy deletion: LOGGED, NOT EXECUTED
   - Recovery disable: LOGGED, NOT EXECUTED
   - Persistence: LOGGED, NOT EXECUTED

3. **Recovery Script**:
   - Automatically deployed during test execution
   - Removes all test artifacts
   - Deletes encrypted files
   - Clears ransom notes from desktop/documents
   - Restores system to pre-test state
