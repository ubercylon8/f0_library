# Defense Guidance: APT34 Exchange Server Weaponization with Email-Based C2

## Executive Summary

This document provides comprehensive defensive guidance against the APT34 (OilRig / Helix Kitten / Hazel Sandstorm) Exchange server weaponization attack chain. This multi-stage attack combines IIS backdoor deployment, email-based command and control, LSA password filter credential interception, and email-based data exfiltration -- representing one of the most sophisticated Exchange compromise methodologies observed in the wild.

## Threat Overview

APT34 is an Iranian state-sponsored threat group specializing in targeting banking, financial services, and government organizations. Their Exchange server weaponization methodology is particularly dangerous because it turns trusted email infrastructure into persistent C2 and exfiltration channels that blend with legitimate traffic.

### Attack Chain

```
[Initial Access / Exchange Compromise]
              |
              v
Stage 1: Deploy CacheHttp.dll IIS Backdoor (T1505.003)
  - Passive HTTP interception via native IIS module
  - Commands via X-Cache-Http headers
  - Responses via cookies
  - Registration via appcmd.exe LOLBin
              |
              v
Stage 2: Establish PowerExchange Email C2 (T1071.003)
  - PowerShell backdoor monitors mailbox via EWS API
  - C2 commands in emails with @@ subject markers
  - Responses sent as email replies
  - Uses svc-exchange@ service account
              |
              v
Stage 3: Register Password Filter DLL (T1556.002)
  - Modify HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Notification Packages
  - DLL intercepts cleartext passwords on every change
  - Persistent credential harvesting (admin, service accounts, krbtgt)
              |
              v
Stage 4: Exfiltrate Data via STEALHOOK (T1048.003)
  - Stage sensitive data (GAL, deal flow, credentials, compliance alerts)
  - Compress into ZIP archive
  - Send as email attachments in multi-part messages (Part N of M)
  - Outbound SMTP to external relay
```

### Why This Attack is Critical for Financial Institutions

- **Exchange servers handle M&A communications** -- compromise enables insider trading
- **Password filter captures all credential rotations** -- persistent domain access
- **Email C2 blends with legitimate traffic** -- extremely difficult to detect via network monitoring
- **STEALHOOK exfiltration mimics normal email** -- bypasses traditional DLP

### Advanced Hardening

#### 1. LSA Hardening (Critical Priority)

**Enable LSA Protected Process (RunAsPPL):**

| Setting | Value |
|---------|-------|
| **Location** | `HKLM\SYSTEM\CurrentControlSet\Control\Lsa` |
| **Key** | `RunAsPPL` |
| **Type** | REG_DWORD |
| **Recommended Value** | 1 |
| **Default Value** | 0 (disabled) |
| **Impact Level** | Medium -- may affect third-party credential providers |

```powershell
# Apply
Set-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa" -Name "RunAsPPL" -Value 1 -Type DWord

# Verify
Get-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa" -Name "RunAsPPL"

# Rollback
Set-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa" -Name "RunAsPPL" -Value 0 -Type DWord
```

**Considerations:**
- Requires reboot to take effect
- Some third-party password filter or credential provider software may be incompatible
- Test in a staging environment before deploying to production domain controllers

#### 2. Exchange-Specific Hardening

**Restrict EWS API Access:**
```powershell
# Limit EWS access to specific applications via Exchange throttling policy
New-ThrottlingPolicy -Name "RestrictEWS" -EwsMaxConcurrency 5 -EwsMaxBurst 10
Set-ThrottlingPolicyAssociation -Identity "svc-exchange" -ThrottlingPolicy "RestrictEWS"

# Block EWS access from non-authorized IPs
New-ClientAccessRule -Name "Block EWS External" `
    -Action DenyAccess `
    -AnyOfProtocols ExchangeWebServices `
    -ExceptAnyOfClientIPAddressesOrRanges "10.0.0.0/8"
```

**Email Content Inspection (DLP Transport Rules):**
```powershell
# Flag @@ subject patterns (PowerExchange C2 indicator)
New-TransportRule -Name "APT34-PowerExchange-C2" `
    -SubjectMatchesPatterns "@@[A-Za-z0-9+/=]+" `
    -SetSCL 9 `
    -Quarantine $true `
    -GenerateIncidentReport "security-team@corp.com"

# Flag sequential exfiltration pattern from service accounts
New-TransportRule -Name "APT34-STEALHOOK-Detection" `
    -FromMemberOf "ServiceAccounts" `
    -AttachmentSizeOver 50KB `
    -SubjectContainsWords "Part","Report" `
    -SetSCL 9 `
    -GenerateIncidentReport "security-team@corp.com" `
    -IncidentReportContent "Sender","Subject","MessageBody"
```

### 1. Detection Triggers

| Detection Name | Trigger Criteria | Confidence | Priority |
|----------------|------------------|------------|----------|
| LSA Password Filter Registration | Notification Packages registry modification | Critical | P1 |
| IIS Backdoor DLL Deployment | CacheHttp.dll or similar DLL in IIS directories | High | P1 |
| PowerExchange C2 Email | Email with @@ subject + base64 payload | Critical | P1 |
| Outbound SMTP from Non-Mail Server | SMTP connection from non-Exchange host | High | P2 |
| Multi-Technique Correlation | 3+ indicators on same host within 2 hours | Critical | P1 |

### Initial Triage Questions

1. Is this a known F0RT1KA test execution or unexpected activity?
2. Is the affected system an Exchange server or IIS host?
3. What service account is associated with the activity?
4. How many stages of the attack chain have been observed?
5. Has any outbound email exfiltration been detected?

### 2. Containment (First 30 Minutes)

- [ ] **Isolate the Exchange server from the network**
  ```powershell
  netsh advfirewall set allprofiles state on
  netsh advfirewall firewall add rule name="IR-Block-Outbound" dir=out action=block
  netsh advfirewall firewall add rule name="IR-Allow-DC" dir=out action=allow remoteip=<DC-IP>
  ```

- [ ] **Remove password filter DLL from LSA**
  ```powershell
  $current = (Get-ItemProperty "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa").'Notification Packages'
  $cleaned = $current | Where-Object { $_ -in @("scecli", "rassfm", "FVENOTIFY", "kdcsvc") }
  Set-ItemProperty "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa" -Name "Notification Packages" -Value $cleaned
  ```

- [ ] **Unregister malicious IIS module**
  ```powershell
  C:\Windows\System32\inetsrv\appcmd.exe uninstall module CacheHttp
  iisreset /restart
  ```

- [ ] **Kill PowerExchange processes**
  ```powershell
  Get-Process powershell, pwsh -ErrorAction SilentlyContinue |
      Where-Object { $_.CommandLine -match "Exchange|EWS|mailbox" } |
      Stop-Process -Force
  ```

- [ ] **Preserve volatile evidence**
  ```powershell
  Get-Process | Export-Csv "C:\IR\processes_$(Get-Date -Format 'yyyyMMdd_HHmmss').csv"
  Get-NetTCPConnection | Export-Csv "C:\IR\connections_$(Get-Date -Format 'yyyyMMdd_HHmmss').csv"
  Copy-Item "$env:SystemRoot\System32\inetsrv\config\applicationHost.config" "C:\IR\"
  ```

### 4. Eradication

```powershell
# Remove IIS backdoor artifacts
Get-ChildItem -Path C:\ -Recurse -Filter "CacheHttp.dll" -ErrorAction SilentlyContinue |
    ForEach-Object { Remove-Item $_.FullName -Force }

# Remove password filter DLL
Get-ChildItem -Path C:\ -Recurse -Filter "*credfilter*" -ErrorAction SilentlyContinue |
    ForEach-Object { Remove-Item $_.FullName -Force }

# Remove PowerExchange artifacts
Get-ChildItem -Path C:\ -Recurse -Filter "PowerExchange*" -ErrorAction SilentlyContinue |
    ForEach-Object { Remove-Item $_.FullName -Force }

# Remove exfiltration staging
Remove-Item -Path "C:\*stealhook*" -Recurse -Force -ErrorAction SilentlyContinue
```

**Credential Rotation (MANDATORY after T1556.002):**
```powershell
# Force password change for all users
Get-ADUser -Filter * | Set-ADUser -ChangePasswordAtLogon $true
# Reset krbtgt twice (with 12-hour gap)
# Reset all service account passwords
```

### 5. Recovery

- [ ] Verify Notification Packages contains only legitimate entries (`scecli`)
- [ ] Verify no unauthorized IIS modules registered
- [ ] Enable LSA protection (RunAsPPL) before bringing server back online
- [ ] Apply all hardening from this document
- [ ] Add mail transport rules to block @@ subject patterns
- [ ] Reconnect to network (after validation)
