# APT34 Exchange Server Weaponization with Email-Based C2

**Category**: APT Simulation / Initial Access / Command and Control
**Severity**: Critical
**MITRE ATT&CK**: T1505.003, T1071.003, T1556.002, T1048.003

## Description

This test simulates the complete attack chain used by APT34 (OilRig / Helix Kitten / Hazel Sandstorm), an Iranian state-sponsored threat group, to weaponize compromised Exchange servers. The simulation covers four critical techniques: deploying a passive IIS backdoor (CacheHttp.dll), establishing email-based C2 via PowerExchange with distinctive "@@" subject markers, registering a password filter DLL in LSA for cleartext credential interception, and exfiltrating stolen data as email attachments via the STEALHOOK tool.

This attack chain is particularly relevant to financial institutions where Exchange servers handle sensitive deal flow, compliance alerts, and client communications.

## Technical Details

### Attack Flow

1. **Stage 1: IIS Backdoor Deployment (T1505.003)**
   - Creates simulated CacheHttp.dll file at c:\F0
   - Generates IIS module registration XML artifact
   - Documents HTTP interception patterns used by the backdoor
   - Detection: File write of DLL in system directories, IIS module registration

2. **Stage 2: Email-Based C2 Channel (T1071.003)**
   - Deploys simulated PowerExchange PowerShell backdoor script
   - Creates 4 C2 command/response email pairs with @@ subject markers
   - Generates C2 channel status documentation
   - Detection: PowerShell script deployment, emails with @@ subjects, EWS API patterns

3. **Stage 3: Password Filter DLL Registration (T1556.002)**
   - Creates benign password filter DLL file
   - Opens HKLM\SYSTEM\CurrentControlSet\Control\Lsa registry key
   - Adds "f0rt1ka_credfilter" to Notification Packages value
   - Verifies registration was successful
   - Detection: LSA registry modification, DLL registration in authentication chain

4. **Stage 4: STEALHOOK Data Exfiltration (T1048.003)**
   - Stages simulated financial/compliance data (GAL, deal flow, credentials)
   - Compresses data into ZIP archive
   - Creates exfiltration emails with base64-encoded attachment chunks
   - Simulates multi-part email exfiltration pattern
   - Detection: Data staging, compression, email attachment generation

### Key Indicators

- **CacheHttp.dll** file creation (IIS backdoor artifact)
- **Emails with "@@" in subject line** (PowerExchange C2 marker)
- **Registry modification**: HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Notification Packages
- **PowerExchange.ps1** script deployment
- **ZIP archive creation** in staging directory
- **Sequential email generation** with "Part N of M" subjects
- **Service account email activity** (svc-exchange@)

## Detection Opportunities

1. **File System Activity**
   - DLL file creation at c:\F0\CacheHttp.dll
   - PowerShell script deployment
   - ZIP archive creation for data staging
   - Multiple email (.eml) file generation

2. **Registry Modification**
   - HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Notification Packages modification
   - Addition of unknown DLL name to password filter list
   - This is a high-fidelity detection point - legitimate changes are rare

3. **Email Pattern Analysis**
   - Emails with "@@" subject markers (C2 command delivery)
   - Sequential "Part N of M" subjects (data exfiltration)
   - Binary attachments from service accounts
   - Automated replies to external addresses

4. **Process Behavior**
   - Stage binary execution from c:\F0
   - Registry access to LSA configuration
   - File compression operations
   - Sequential multi-stage execution pattern

## Expected Results

### Unprotected System (Code 101)
All four stages complete successfully:
- CacheHttp.dll backdoor deployed
- PowerExchange C2 channel established with email artifacts
- Password filter DLL registered in LSA (f0rt1ka_credfilter added to Notification Packages)
- STEALHOOK exfiltration simulation completed with staged data and email artifacts

### Protected System (Enhanced Detection)
- **Code 105**: Stage binary quarantined before execution (AV signature match)
- **Code 126**: Stage blocked during execution (registry write denied, file creation blocked, or behavioral detection triggered)

Most likely detection points:
1. Stage 3 (T1556.002) - LSA registry modification is a high-confidence EDR detection
2. Stage 1 (T1505.003) - DLL file with PE header markers may trigger heuristic detection
3. Stage 4 (T1048.003) - Data staging and compression patterns
