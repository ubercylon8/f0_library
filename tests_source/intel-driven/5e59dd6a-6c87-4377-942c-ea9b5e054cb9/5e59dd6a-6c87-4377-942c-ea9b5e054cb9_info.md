## Purpose

Give EDR/AV products a safe, controlled workload that exercises the observable API surface of BlueHammer's initial phases (1-3). The goal is to let defenders measure detection coverage on the *detectable primitives* — the ones that appear BEFORE the credential-access phases an EDR must prevent to stop BlueHammer end-to-end.

This test does NOT simulate credential theft, SAM access, or any later phase of the PoC. Those phases are described below for documentation only.

## MITRE mapping rationale

**T1211 — Exploitation for Defense Evasion** is mapped because the test exercises two primitives that the original PoC uses against Defender's scanning surface:

1. Cloud Files sync-root registration with a fetch-placeholder callback — the real PoC uses this to intercept Defender's scan path and freeze it.
2. Transacted-file-open + VSS device enumeration — the real PoC uses this to win a TOCTOU race against Defender's VSS handle lifecycle.

Both are detection-evasion primitives; we exercise them without the downstream exploit payload.

**T1562.001 — Impair Defenses: Disable or Modify Tools** is mapped because the batch-oplock primitive (`FSCTL_REQUEST_BATCH_OPLOCK` on a file in Defender's scan path) is a known scanner-stall technique. We exercise it against a sandbox file — not a file Defender actually scans — so no defense is actually impaired. But the API-level behavior is the same telemetry pattern an EDR would see during a real attempt.

**T1003.002 (Credential Dumping: SAM) was DELIBERATELY EXCLUDED.** The real BlueHammer PoC culminates in SAM database access, but this F0RT1KA test never touches SAM, never opens a VSS path, never loads offreg or samlib.dll, and never reads any credential material. Claiming T1003.002 coverage from a test that doesn't execute any SAM-adjacent API would be a false-positive coverage claim that misrepresents what this test actually proves. If a defender wants T1003.002 coverage they should deploy an explicit SAM-dump test — this test explicitly avoids that API surface by design.

## Detailed stage behavior

### Stage 1 — `T1211-cfapi` (Cloud Files sync-root + fetch-placeholder + EICAR)

1. Creates `C:\Users\fortika-test\BlueHammerSandbox\cfapi-root\` (sandbox, non-whitelisted)
2. Calls `CfRegisterSyncRoot(sandboxPath, {ProviderName="F0RT1KA-BLUEHAMMER-SIM", ProviderVersion="1.0"}, policies, CF_REGISTER_FLAG_NONE)`
3. Calls `CfConnectSyncRoot(sandboxPath, callbacks[CF_CALLBACK_TYPE_FETCH_PLACEHOLDERS -> no-op Go callback], ...)`
4. Drops the EICAR test string to `cfapi-root\eicar-probe.txt` (plain forward drop; no obfuscation)
5. Sleeps 3 seconds, runs `cmd /c dir <sandbox>` to surface a listing event
6. `CfDisconnectSyncRoot` + `CfUnregisterSyncRoot` cleanup unconditionally

**Telemetry surface**: Cloud Files API usage by a non-vendor provider, EICAR drop in a provider-registered directory, a fetch-placeholder callback registration by a process that isn't OneDrive/iCloud/Google Drive.

### Stage 2 — `T1562.001-oplock` (Batch oplock on sandbox file)

1. Creates `C:\Users\fortika-test\BlueHammerSandbox\oplock-target.txt` (sandbox file, 68-byte benign seed content)
2. Opens the file with `CreateFile(GENERIC_READ|GENERIC_WRITE, FILE_SHARE_READ, OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL|FILE_FLAG_OVERLAPPED)`
3. Calls `DeviceIoControl(handle, FSCTL_REQUEST_BATCH_OPLOCK (0x00090018), OVERLAPPED{hEvent})` — the request returns `ERROR_IO_PENDING` on the happy path
4. Waits `WaitForSingleObject(hEvent, 5000ms)` — hard 5-second cap
5. If the oplock broke: logs the transferred-bytes count; if it timed out: calls `CancelIoEx(handle, &ov)`
6. Closes the handle (via `defer`) — releasing the oplock regardless of state
7. Deletes the sandbox file

**Telemetry surface**: `FSCTL_REQUEST_BATCH_OPLOCK` on a file from a non-system process, combined with `FILE_FLAG_OVERLAPPED` and a 5-second wait.

**Safety**: the target is the test's own sandbox file. Defender does not scan it in a way that would interact with the oplock, so no actual freeze is produced — this is a pure API-surface exercise.

### Stage 3 — `T1211-vssenum` (VSS enumeration + transacted-open on sandbox file)

**Part A** — VSS device enumeration (read-only):
1. `NtOpenDirectoryObject(\Device, DIRECTORY_QUERY)` — standard-user permitted
2. `NtQueryDirectoryObject` loop, reading up to 64 KB of entries
3. For each entry, the name is materialized from the UTF-16 buffer and checked with `hasPrefix("HarddiskVolumeShadowCopy")`
4. Matches are counted and logged — no matched entry is ever opened
5. `NtClose(hDir)` cleanup

**Part B** — Transacted-open probe on a sandbox file:
1. Creates `C:\Users\fortika-test\BlueHammerSandbox\transacted-probe.txt` (benign 68-byte seed)
2. Calls `CreateTransaction(NULL, NULL, 0, NULL, NULL, NULL, NULL)` — creates a KTM transaction
3. Calls `CreateFileTransactedW(probePath, GENERIC_READ, FILE_SHARE_READ, NULL, OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL, NULL, hTransaction, NULL, NULL)`
4. If a handle is returned, it is `CloseHandle`'d immediately
5. `RollbackTransaction(hTransaction)` + `CloseHandle(hTransaction)` via `defer` — transaction is NEVER committed
6. Deletes the sandbox file

**Telemetry surface**: `NtOpenDirectoryObject(\Device)` from a non-system process, `CreateFileTransactedW` usage (a rarely-used API outside of installer/ktm scenarios), transaction rollback without commit.

**Safety**: the transacted-open target is always the sandbox file, never a VSS path, never a system file.
