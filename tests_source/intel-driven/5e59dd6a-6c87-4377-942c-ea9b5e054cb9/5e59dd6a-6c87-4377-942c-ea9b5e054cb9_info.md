## Purpose

Give EDR/AV products a safe, controlled workload that exercises the observable API surface of BlueHammer's full kill-chain primitives â€” including the SAM-hive read API surface that BlueHammer's PoC ultimately uses, but executed against a **synthetic sandbox hive** rather than any real Windows hive. The goal is to let defenders measure detection coverage on every primitive BlueHammer touches, with strict safety guarantees ensuring no real credential material is ever accessed.

## MITRE mapping rationale

**T1211 â€” Exploitation for Defense Evasion** (stages 1, 3 Part A)
- Cloud Files sync-root registration with a fetch-placeholder callback (PoC uses this to intercept Defender's scan path)
- NT Object Manager enumeration of `\Device` for VSS shadow copies (PoC uses this to find a snapshot to attack)

**T1562.001 â€” Impair Defenses: Disable or Modify Tools** (stage 2 + stage 3 Part B)
- `FSCTL_REQUEST_BATCH_OPLOCK` on a file (PoC uses against a Defender scan path; we use against a sandbox file)
- Read-only WMI Win32_ShadowCopy enumeration (PoC reconnoiters before VSS abuse; we exercise the WQL signal without proceeding)

**T1003.002 â€” OS Credential Dumping: Security Account Manager** (stage 4)
- `RegLoadKey` + `RegOpenKeyEx(KEY_READ)` â€” the API signature BlueHammer uses to read SAM. We execute this against a **synthetic sandbox hive we generated at runtime**, never against a real SAM hive. The detection telemetry an EDR sees is identical; the credential exposure is zero.

**T1134.001 â€” Access Token Manipulation: Token Impersonation/Theft** (stage 4)
- `AdjustTokenPrivileges` enabling `SeBackupPrivilege` and `SeRestorePrivilege` â€” the privilege-enable telemetry signal that precedes any SAM-hive load attempt.

## Detailed stage behavior

### Stage 1 â€” `T1211-cfapi` (Cloud Files sync-root + Mimikatz-named EICAR drop)

1. Creates `C:\Users\fortika-test\BlueHammerSandbox\cfapi-root\`
2. `CfRegisterSyncRoot(sandboxPath, {ProviderName="F0RT1KA-BLUEHAMMER-SIM", ProviderVersion="1.0"}, policies, CF_REGISTER_FLAG_NONE)`
3. `CfConnectSyncRoot(sandboxPath, callbacks[CF_CALLBACK_TYPE_FETCH_PLACEHOLDERS -> no-op Go callback], ...)`
4. **(B2)** Drops EICAR test string to `Mimikatz_dump_Win32_signature.txt` â€” content is EICAR (AMSI signal); filename suggests Mimikatz (path-rule signal). Both detection paths fire.
5. Sleep 3s, `cmd /c dir <sandbox>` to surface a listing event
6. `CfDisconnectSyncRoot` + `CfUnregisterSyncRoot` cleanup unconditionally

### Stage 2 â€” `T1562.001-oplock` (Batch oplock on sandbox file)

Unchanged from v1. `FSCTL_REQUEST_BATCH_OPLOCK` on a sandbox file with hard 5-second timeout; oplock always released regardless of break/timeout.

### Stage 3 â€” `T1211-vssenum` (NT enum + WMI enum + transacted-open)

Part A unchanged (NT Object Manager enumeration of `\Device`).

**(B3) Part B is new:** WMI Win32_ShadowCopy enumeration via PowerShell `Get-WmiObject -Class Win32_ShadowCopy`. The query is purely enumerative â€” `Win32_ShadowCopy.Create()` and `.Delete()` are NEVER called. PoC fidelity deviation: the real PoC uses direct `IWbemServices->ExecQuery` COM; this test invokes the same provider via PowerShell for implementation simplicity. The sensor-visible event is identical.

Part C (transacted-open probe on sandbox file) unchanged.

### Stage 4 â€” `T1003.002-samsim` (Privilege-enable + synthetic SAM-hive load) **NEW in v2**

1. **(B4)** `OpenProcessToken(GetCurrentProcess(), TOKEN_ADJUST_PRIVILEGES|TOKEN_QUERY)`
2. **(B4)** `LookupPrivilegeValueW("SeBackupPrivilege")` + `AdjustTokenPrivileges(SeBackupPrivilege, ENABLED)` â€” telemetry regardless of OS grant/deny
3. **(B4)** Same for `SeRestorePrivilege` â€” required to proceed to part B
4. **Standard-user / non-admin path:** if `SeRestorePrivilege` was not granted, the OS returns `ERROR_NOT_ALL_ASSIGNED`. The privilege-denial telemetry signal is logged and the stage exits success (B4 was the value). **No hive load is attempted.**
5. **Admin / elevated path:** stage proceeds to part B:
   - **(B5)** `RegCreateKeyEx(HKLM\SOFTWARE\F0_BLUEHAMMER_SAMSIM_TEMP)` â€” creates a sandbox temp key
   - **(B5)** `RegSaveKey(tempKey, ARTIFACT_DIR\sam-sim.hiv)` â€” writes a synthetic hive file
   - **(B5)** Watchdog goroutine starts (30s timeout). If the hive remains loaded past timeout, it is force-unloaded.
   - **(B5)** `RegLoadKey(HKLM, "F0_BLUEHAMMER_SAMSIM", sam-sim.hiv)` â€” loads the SYNTHETIC sandbox hive under a unique mount name
   - **(B5)** `RegOpenKeyEx(HKLM\F0_BLUEHAMMER_SAMSIM, KEY_READ)` â€” the SAM-hive-read API signature
   - `RegCloseKey` + `RegUnLoadKey` (defer LIFO order)
   - Watchdog stop + defer cleanup of temp HKLM key + sandbox file

**Safety guarantees for stage 4:**
- The hive loaded under HKLM\F0_BLUEHAMMER_SAMSIM is a **synthetic sandbox file** the test created, not any real Windows hive
- The mount name is unique to this test (contains "BLUEHAMMER")
- Watchdog (30s) force-unloads the hive if any subsequent step hangs
- Temp HKLM key is removed on cleanup (defer, LIFO)
- Sandbox hive file is removed on cleanup (defer)
- Standard-user execution skips part B entirely

## Lab-Bound Observability (added 2026-04-25 after lab evidence)

Lab runs against a Windows endpoint with stock Microsoft Defender + MDE confirmed that **the kill chain breaks at stage 2**: Defender's cloud-delivered content-based detection on the T1562.001-oplock stage binary quarantines it ~2 minutes after extraction (during stage 1's runtime). The detection survives dual-signing with the sb org cert + F0RT1KA nested signature â€” it is technique-leaning, not artifact-leaning, and matches on the binary's `FSCTL_REQUEST_BATCH_OPLOCK` IOCTL constant or the oplock API import sequence rather than on unsigned-PE characteristics.

Consequence: stages 3 (T1211-vssenum + WMI shadow enum) and 4 (T1003.002-samsim) are **lab-unobservable on this sensor stack**. The orchestrator's stage-2 quarantine handler maps the missing binary to exit 105 (FileQuarantinedOnExtraction), the test correctly reports overall exit 126 (Protected), and stages 3+4 are recorded as `skipped` in the bundle.

This is **not a test bug** and does **not warrant refactoring**. Three reasons:

1. **The current implementation is faithful to the published Nightmare-Eclipse PoC.** The PoC ships static IOCTL constants and unobfuscated API calls. A real attacker running BlueHammer in this form would also be stopped at stage 2. The test's behavior matches reality.

2. **Refactoring stage 2 to evade Defender's content match would lower API Fidelity.** Obfuscating the IOCTL constant, switching to indirect syscalls via `NtFsControlFile`, or breaking into a loader+payload would simulate a *different* attacker â€” an "EDR-aware APT" rather than the PoC's default behavior. That trade-off (higher Detection Firing Fidelity, lower API Fidelity) is a real rubric tension worth preserving as a deliberate decision rather than an incidental refactor.

3. **The detection-coverage signal is real and useful.** "Defender catches the kill chain at the oplock primitive" is itself the answer to the test's question. Proving downstream stages 3, 4 are *individually* detectable is a different question, more appropriately answered by single-primitive companion tests (e.g., a standalone WMI Win32_ShadowCopy enumeration test, a standalone synthetic-SAM-load test) rather than by reshaping BlueHammer.

### Implications for v2 Detection Firing Fidelity scoring

Detection Firing Fidelity (sub-dimension 2c) is bounded by what's lab-observable. For BlueHammer specifically:

- **Stage 1 primitives** (cfapi, fetch-placeholder callback, Mimikatz-EICAR drop): observable. Lab evidence: stage 1 ran clean, no Defender block at the stage level.
- **Stage 2 primitives** (oplock IOCTL): partially observable. Lab evidence: Defender's content-based detection fires on the binary itself (not on the runtime IOCTL call). Counts as detection of the technique surface, but with the caveat that it's a static-binary signature not a behavioral rule.
- **Stage 3 primitives** (NT \Device enum, WMI Win32_ShadowCopy enum): unobservable on this stack. Would require either (a) Defender exclusion for the stage-2 binary or (b) a single-primitive companion test.
- **Stage 4 primitives** (SeBackupPrivilege/SeRestorePrivilege, RegLoadKey, RegOpenKey on loaded hive): unobservable on this stack. Same options.

The v2 Detection Firing score for BlueHammer therefore caps not just at 0.5 (pre-LC-rule-deployment) but at whatever fraction of the rule corpus targets stages 1+2 specifically â€” even with full LC rule deployment, rules targeting stages 3+4 will not fire in this lab without one of the workarounds above.

This is a **shape-of-kill-chain** issue, not a test-quality issue. It applies to any multi-stage test where an early stage is reliably caught: the downstream stages become unmeasurable in that lab. Worth recognizing as a rubric design constraint rather than driving test refactors around it.
