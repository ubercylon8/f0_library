## Threat Context

**UnDefend** was published by Nightmare-Eclipse as a PoC demonstrating that a
**standard-user process** (no admin required, no driver, no handle hijack) can
indefinitely disable Microsoft Defender signature and engine updates by racing
Windows' own file replace operations.

### Attack Preconditions

| Precondition | Required |
|---|---|
| Code execution as any user (SYSTEM not required) | Yes |
| Read access to HKLM\SOFTWARE\Microsoft\Windows Defender | Yes (standard-user default) |
| Read access to `C:\ProgramData\Microsoft\Windows Defender\Definition Updates` | Yes (standard-user default) |
| Ability to call `CreateFile` / `NtCreateFile` | Yes (universal) |
| Admin / SYSTEM / TrustedInstaller | **No** |

### Real-World Impact

A successful UnDefend run leaves the endpoint with:
- Stale Defender signature database (no new-threat detection coverage)
- Silent signature-update failures (no obvious error in Defender UI)
- During major platform updates, a non-functional engine (`mpavbase.vdm` locked)

## Simulation Architecture

### Why Multi-Stage

UnDefend's three attack vectors are distinct detection surfaces:
- **Stage 1 (T1083)**: Registry + directory watcher recon — visible at registry-read and FS-handle-open time
- **Stage 2 (T1562.001)**: File-lock primitive — visible at `CreateFile`/`LockFile`/`LockFileEx` time
- **Stage 3 (T1562.001)**: Service-notification subscription — visible at `NotifyServiceStatusChangeW` time

A monolithic binary would conflate these signals; a multi-stage architecture gives
detection-engineering precision (each stage is independently observable and
attributable via `bundle_results.json`).

### Stage Details

**Stage 1 — Defender Update-Path Discovery (T1083)**
1. Opens `HKLM\SOFTWARE\Microsoft\Windows Defender`, reads `ProductAppDataPath`.
2. Opens `HKLM\SOFTWARE\Microsoft\Windows Defender\Signature Updates`, reads `SignatureLocation`.
3. Calls `CreateFile(DefinitionUpdatesDir, FILE_LIST_DIRECTORY|SYNCHRONIZE, ..., FILE_FLAG_BACKUP_SEMANTICS)`.
4. Calls `ReadDirectoryChangesW(handle, ..., recursive=true, FILE_NOTIFY_CHANGE_SIZE, &ov)`.
5. Waits up to 2s for a change notification; cancels I/O and closes handle.

**Stage 2 — UnDefend File-Lock Primitive (T1562.001)**
1. Creates `ARTIFACT_DIR\UnDefendSim\sig-sample.yaml` (benign content).
2. Opens it with `CreateFile(GENERIC_READ|SYNCHRONIZE, FILE_SHARE_WRITE|FILE_SHARE_DELETE)`.
3. Calls `LockFile(handle, 0, 0, sizeLow, sizeHigh)` — acquires exclusive byte-range lock.
4. Releases with `UnlockFile`.
5. Re-opens and invokes `LockFileEx(handle, LOCKFILE_EXCLUSIVE_LOCK, 0, sizeLow, sizeHigh, &ov)`.
6. Releases with `UnlockFileEx`, deletes the sandbox file.

**Stage 3 — WinDefend Service-Stop Notification Subscription (T1562.001)**
1. `OpenSCManager(NULL, NULL, SC_MANAGER_CONNECT)`.
2. `OpenService(scm, "WinDefend", SERVICE_QUERY_STATUS|SERVICE_QUERY_CONFIG)`.
3. `QueryServiceStatus` — confirms `SERVICE_RUNNING`.
4. `NotifyServiceStatusChangeW(svc, SERVICE_NOTIFY_STOPPED, &notify)`.
5. Returns immediately (no `SleepEx` — callback guaranteed not to fire).
6. Subscription torn down when service handle closes at function return.

## MITRE ATT&CK Mapping

| Technique | Tactic | Why |
|---|---|---|
| [T1562.001 — Impair Defenses: Disable or Modify Tools](https://attack.mitre.org/techniques/T1562/001/) | defense-evasion | The `LockFile`/`LockFileEx` primitive (stage 2) and the `WinDefend` service-stop subscription (stage 3) are both components of UnDefend's Defender-disabling chain. |
| [T1083 — File and Directory Discovery](https://attack.mitre.org/techniques/T1083/) | discovery | Stage 1 reads Defender registry paths and subscribes to directory changes on `Definition Updates` — classic reconnaissance of a security product's operational filesystem. |

### Techniques Considered and Rejected

- **T1489 (Service Stop)** — UnDefend does not stop any service; it only *subscribes* to stop notifications. Mapping rejected.
- **T1055 (Process Injection)** — no injection of any kind. Rejected.
- **T1112 (Modify Registry)** — only registry READS, no writes. Rejected.

## Detection Summary

See `6a2351ac-654a-4112-b378-e6919beef70d_detections.kql`,
`_rules.yar`, `_sigma_rules.yml`, `_elastic_rules.ndjson`, `_dr_rules.yaml`.

Highest-fidelity behavioral signals:
1. **Non-admin process calling `NotifyServiceStatusChangeW` on `WinDefend`** — this pattern has essentially no legitimate use outside of security products themselves.
2. **`ReadDirectoryChangesW` handle opened on `ProgramData\Microsoft\Windows Defender\Definition Updates`** by a non-Microsoft-signed process.
3. **Registry queries for `ProductAppDataPath` + `SignatureLocation` under `HKLM\SOFTWARE\Microsoft\Windows Defender`** from a process that subsequently calls `LockFile`/`LockFileEx`.

## Defense Summary

See `6a2351ac-654a-4112-b378-e6919beef70d_DEFENSE_GUIDANCE.md` and
`6a2351ac-654a-4112-b378-e6919beef70d_hardening.ps1`.

Key controls:
- Enable Defender tamper protection (attack still works without admin, but tamper protection blocks related escalation).
- ASR rule: Block untrusted and unsigned processes that run from USB (and general ASR hardening).
- Monitor process→Defender-service interactions (EDR custom rule).
- Accelerated signature-update failure alerting (Defender has ETW events for failed updates).
