## Description

Simulates the PROMPTFLUX malware family disclosed by the Google Threat Intelligence Group (GTIG) in November 2025. PROMPTFLUX is a VBScript dropper whose defining characteristic is **runtime LLM abuse** — it calls the Google Gemini API at regular intervals (reportedly hourly) via a module named **"Thinging"** and asks the model to produce a new variant of itself. The returned code overwrites the on-disk VBS in place, giving the malware a metamorphic property without a custom packer: each host ends up with a distinct content hash for the same semantic behaviour.

PROMPTFLUX's additional on-host tradecraft is otherwise commodity:
- Dropper lives at `c:\F0\crypted_ScreenRec_webinstall.vbs`
- Staging trail at `%TEMP%\thinking_robot_log.txt` (raw LLM response)
- Persistence via a `.vbs` dropped into the user Startup folder
- Propagation by enumerating removable and network drives via WMI (`Win32_LogicalDisk`) and copying itself to each

This test reproduces the **shape** of each of those behaviours across four stages so EDR/NDR teams can develop behavioural detections. The VBS bodies are **benign** — they only echo marker strings and drop marker files — and the propagation step is **enumeration-only** (no copy).

## Technical Details

### Attack Flow

1. **Stage 1: LLM API Fetch (T1071.001)**
   - HTTPS GET to `https://raw.githubusercontent.com/projectachilles/ProjectAchilles/main/lab-assets/promptflux/v1/gemini_response.json`
   - UA: `PromptfluxClient/1.0 (simulated)` — YARA-matchable string IOC
   - Parses response envelope; extracts `candidates[0].content.parts[0].text`
   - Writes raw response to `%TEMP%\thinking_robot_log.txt` (GTIG-documented staging trail)
   - Writes VBS body to `c:\F0\crypted_ScreenRec_webinstall.vbs` (GTIG-documented lure filename)
   - Invokes `wscript.exe //Nologo c:\F0\crypted_ScreenRec_webinstall.vbs`
   - Captures wscript stdout/stderr to `c:\F0\stage1_wscript_output.txt`
   - Confirms execution via `c:\F0\promptflux_stage1_marker.txt`

2. **Stage 2: Metamorphic Rewrite (T1027.001 — Thinging)**
   - Snapshots SHA256 of the stage-1 VBS
   - HTTPS GET to `https://raw.githubusercontent.com/projectachilles/ProjectAchilles/main/lab-assets/promptflux/v1/variant_thinging.vbs`
   - Confirms new content hash differs from pre-rewrite hash (guards against lab-asset rotation failure)
   - Overwrites `c:\F0\crypted_ScreenRec_webinstall.vbs` with new body in place
   - Invokes `wscript.exe //Nologo c:\F0\crypted_ScreenRec_webinstall.vbs` on the rewritten file
   - Confirms execution via `c:\F0\promptflux_stage2_marker.txt`

3. **Stage 3: Startup Folder Persistence (T1547.001)**
   - Resolves Startup folder based on execution context:
     - User context → `%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup`
     - SYSTEM context → `%ProgramData%\Microsoft\Windows\Start Menu\Programs\StartUp`
   - Writes embedded third obfuscated benign VBS as `ScreenRecUpdater.vbs`
   - **Does NOT execute** — execution at next logon is the modeled TA behaviour; the file-on-disk is the detection signal at test-time
   - Orchestrator's `cleanupAll()` removes this file on every exit path

4. **Stage 4: Propagation Enumeration (T1091)**
   - Invokes `wmic.exe logicaldisk get DeviceID,DriveType,VolumeName /format:csv`
   - Parses CSV output; filters DriveType=2 (removable) and DriveType=4 (network)
   - Writes structured JSON report to `c:\F0\stage4_propagation_targets.json`
   - **No copy** — deliberately stops at enumeration per test safety policy
   - Permission-denied / empty-output from wmic = exit 126 (defensive posture detected)

### Key Indicators

- Real TLS handshake against GitHub's fleet certificate (genuine JA3/JA4 fingerprint)
- DNS query for `raw.githubusercontent.com` from `c:\F0\<uuid>-T1071.001.exe` and `c:\F0\<uuid>-T1027.001.exe`
- HTTP GET with User-Agent `PromptfluxClient/1.0 (simulated)` or `PromptfluxClient/1.0 (simulated thinging)`
- Creation of `c:\F0\crypted_ScreenRec_webinstall.vbs` followed by `wscript.exe` child with that path on command line
- Creation of `%TEMP%\thinking_robot_log.txt` by a non-browser, non-Office process
- In-place overwrite of `c:\F0\crypted_ScreenRec_webinstall.vbs` with changed content hash within ~10 seconds of creation
- VBS file written to any `Start Menu\Programs\Startup` folder with filename `ScreenRecUpdater.vbs`
- `wmic.exe` spawned with `logicaldisk` argument by a binary living in `c:\F0`

## Detection Opportunities

1. **LLM-proxy fetch** — Non-browser process issuing HTTPS GET to `raw.githubusercontent.com/projectachilles/ProjectAchilles/*/lab-assets/promptflux/*`. Extend to any GitHub raw path from a binary in `c:\F0\` for a high-signal behavioural rule.
2. **wscript staging from user data dir** — `wscript.exe` child with first argument pointing into `c:\F0\*.vbs`, parent binary unsigned or signed by a non-Microsoft issuer.
3. **Thinging metamorphism** — Same `c:\F0\*.vbs` filename observed as `File Created` then `File Modified` with different content hashes within 60 seconds, both by the same non-Office parent process.
4. **Thinking trail artefact** — File creation event for `%TEMP%\thinking_robot_log.txt` by any process other than `notepad.exe`, `explorer.exe`, or developer tooling.
5. **Startup folder VBS drop** — File creation of any `.vbs` under `Start Menu\Programs\Startup` folders by a non-installer (not MSI/setup/update tooling). Filename `ScreenRecUpdater.vbs` as a high-confidence string IOC.
6. **WMI propagation enumeration** — `wmic.exe` child with `logicaldisk` argument, parent binary in `c:\F0\` or unsigned.
7. **Combined correlation** — Host that produces two or more of the above within a 5-minute window. This is the killchain rule for mature SIEMs.

## Expected Results

### Unprotected System (Code 101)

- Stages 1, 2, 3, 4 complete successfully
- `c:\F0\crypted_ScreenRec_webinstall.vbs` present and wscript-executed (twice, with different hashes)
- `%TEMP%\thinking_robot_log.txt` present
- Stage-1 and stage-2 markers present in `c:\F0`
- Stage-3 `ScreenRecUpdater.vbs` present in Startup folder
- Stage-4 `stage4_propagation_targets.json` lists enumerated removable/network drives
- **Impact**: full PROMPTFLUX on-host behavioural chain executed without prevention

### Protected System (Enhanced Detection)

- **Code 105**: Stage binary quarantined at extraction (signature-based AV caught the PE)
- **Code 126**: Stage blocked during execution (behavioural EDR rule fired):
  - Stage 1 blocked: HTTPS fetch blocked, or wscript.exe launch prevented, or VBS file quarantined at write
  - Stage 2 blocked: variant fetch blocked, or in-place overwrite blocked (file-integrity rule), or second wscript invocation prevented
  - Stage 3 blocked: write to Startup folder denied by ACL/ASR
  - Stage 4 blocked: wmic.exe logicaldisk query denied, or empty output (policy-blocked WMI)

## Cleanup

The orchestrator handles cleanup unconditionally on every exit path:
- Normal return → `cleanupAll("unprotected"|"blocked")`
- Panic → `recover()` inside `defer` → `cleanupAll("panic")` → SaveLog → Endpoint.Stop
- SIGINT/SIGTERM → signal handler goroutine → `cleanupAll("signal")` → os.Exit
