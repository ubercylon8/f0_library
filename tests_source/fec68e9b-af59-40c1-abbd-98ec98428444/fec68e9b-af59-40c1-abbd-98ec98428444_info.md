## Description

This test replicates production-accurate MDE exploitation techniques discovered by InfoGuard Labs, performing **actual process injection attempts** against the running MsSense.exe process, **real memory patching** of CRYPT32 cryptographic functions, and **live API command interception** using race conditions against production endpoints.

Unlike simulation-based tests, this performs genuine exploitation techniques:
- Real process handle acquisition with escalating privileges
- Actual memory enumeration and manipulation of MsSense.exe
- Live memory patching of `CRYPT32!CertVerifyCertificateChainPolicy`
- API race conditions to steal commands before legitimate agent
- Unauthenticated token generation and configuration exfiltration

This test represents the **highest-fidelity MDE security assessment** in the F0RT1KA framework, providing comprehensive evaluation of both **process protection** and **API authentication** security controls.

## Technical Details

### Attack Flow

1. **Phase 1: Prerequisites & Environment Validation**
   - Initializes comprehensive test_logger for forensic tracking
   - Extracts embedded watchdog binary to C:\F0\mde_process_watchdog.exe
   - Extracts emergency recovery PowerShell script
   - **CRITICAL CHECK**: Verifies MDE is installed (fails if MsSense.exe not found)
   - **CRITICAL CHECK**: Verifies administrator privileges (required for process injection)
   - Detects Windows version, Defender status, process architecture
   - Logs system configuration for baseline

2. **Phase 2: MDE Process Discovery**
   - Enumerates running processes to locate MsSense.exe
   - Extracts process details: PID, parent PID, executable path, command line
   - Detects SenseIR.exe if present (Live Response agent)
   - Extracts real MDE identifiers from registry (Machine ID, Tenant ID, Sense ID)
   - Enumerates loaded modules in MsSense.exe process
   - Locates CRYPT32.dll base address for Phase 4
   - Detects process architecture (x64/x86) for correct injection
   - Logs all process information with masked sensitive values

3. **Phase 3: Process Handle Acquisition Attempts**
   - **Test 1**: Attempt `OpenProcess` with `PROCESS_VM_READ` (read-only access)
     - Expected: May succeed on unprotected systems (needed for legitimate debugging)
     - Log: Handle value or access denied error
   - **Test 2**: Attempt `OpenProcess` with `PROCESS_VM_WRITE` + `PROCESS_VM_OPERATION`
     - Expected: EDR should BLOCK this on protected systems
     - Log: Success (VULNERABLE) or Access Denied (PROTECTED)
   - **Test 3**: Attempt `OpenProcess` with `PROCESS_CREATE_THREAD` (for injection)
     - Expected: EDR should BLOCK this
     - Log: Success or failure with error codes
   - **Detection Point**: EDR should deny privileged handle acquisition to its own process

4. **Phase 4: Memory Enumeration & CRYPT32 Analysis**
   - If handle acquired in Phase 3, enumerate memory regions using `VirtualQueryEx`
   - Locate CRYPT32.dll module in MsSense.exe address space
   - Resolve `CertVerifyCertificateChainPolicy` function address
   - Read function prologue (first 20 bytes) using `ReadProcessMemory`
   - Analyze function structure to verify patch target
   - Log memory layout, module base addresses, function locations
   - **Detection Point**: Memory scanning of security agent process

5. **Phase 5: Memory Patching Attempt (with Watchdog)**
   - **Watchdog Start**: Launch mde_process_watchdog.exe monitoring this test
   - Store original function bytes (first 5 bytes) for restoration
   - Prepare patch: `33 C0 40 C3` (xor eax,eax; inc eax; ret) - always return TRUE
   - Attempt `WriteProcessMemory` to `CertVerifyCertificateChainPolicy`
   - **If successful**: Set 30-second auto-rollback timer
   - **If blocked**: Log protection mechanism (error code, failure reason)
   - Watchdog monitors for test crash/timeout and auto-restores if needed
   - **Detection Point**: Memory manipulation of critical cryptographic functions

6. **Phase 6: Network Proxy Configuration**
   - Test `netsh winhttp set proxy 127.0.0.1:8080` for Burp-style interception
   - Monitor if MsSense.exe respects proxy settings
   - Check for certificate validation errors in Windows event logs
   - Verify if cert pinning prevents proxied connections
   - Restore original proxy configuration
   - Log all proxy-related events and errors
   - **Detection Point**: Suspicious network configuration changes targeting EDR

7. **Phase 7: API Command Interception (Race Condition)**
   - Spawn background goroutine continuously polling `/edr/commands/cnc`
   - Send rapid unauthenticated requests using real Machine ID and Tenant ID
   - Race legitimate MsSense.exe agent to steal pending commands
   - Commands include: `isolationcommand`, `scancommand`, `forensicscollectioncommand`
   - If command stolen, legitimate agent receives empty response
   - Log intercepted commands, timing, success/failure
   - **Detection Point**: Unauthenticated API requests and command theft patterns

8. **Phase 8: CloudLR Token Generation Bypass**
   - Send unauthenticated request to `/senseir/v1/actions/` endpoint
   - Attempt to obtain valid CloudLR (Cloud Live Response) token
   - Tokens grant capabilities: `command_execution`, `file_download`, `file_upload`
   - Test if token is accepted for Live Response operations
   - Log token structure, capabilities, expiration
   - **Detection Point**: Unauthorized Live Response token generation

9. **Phase 9: Isolation Command Spoofing**
   - If isolation command intercepted in Phase 7, prepare spoofed response
   - Send fake "Already isolated" status to `/commands/status` endpoint
   - Verify device still maintains full network connectivity
   - Device appears isolated in Microsoft Defender Portal but remains operational
   - Creates false sense of security for SOC teams
   - Log isolation status discrepancy
   - **Detection Point**: Isolation bypass with spoofed telemetry

10. **Phase 10: Configuration Exfiltration**
    - Send unauthenticated request to `/edr/commands/cnc` endpoint
    - Retrieve 8MB configuration file (no authentication required)
    - Parse configuration for detection rules, exclusions, monitored processes
    - Extract valuable information:
      - Detection rule patterns (RULE_POWERSHELL_ENCODED_COMMAND, etc.)
      - Process exclusions (paths, process names, extensions)
      - Monitored processes and behaviors
      - Certificate pins and cloud endpoints
    - Save exfiltrated config to C:\F0\exfiltrated_config.json
    - **Detection Point**: Large unauthenticated configuration file retrieval

11. **Phase 11: Final Verification & Cleanup**
    - Wait 10 seconds for delayed EDR behavioral detection
    - Check if any components were quarantined during execution
    - Verify watchdog has restored all memory patches
    - Ensure all process handles are closed
    - Generate comprehensive attack summary covering all 11 phases
    - Save complete execution logs (JSON + text formats)
    - Determine final test outcome based on all phase results
    - Clean up temporary files and restore system state

## Detection Opportunities

### 1. Process Enumeration
**Alert On**: Suspicious enumeration of MsSense.exe or SenseIR.exe processes
```
DeviceProcessEvents
| where ProcessCommandLine contains "MsSense.exe" or ProcessCommandLine contains "SenseIR.exe"
| where InitiatingProcessFileName !in ("MsSense.exe", "SenseIR.exe", "SenseCncProxy.exe")
| where ActionType == "ProcessEnumeration"
```

### 2. Privileged Handle Acquisition
**Alert On**: Attempts to open handles to EDR processes with injection privileges
```
DeviceEvents
| where ActionType == "OpenProcessApiCall"
| where FileName in ("MsSense.exe", "SenseIR.exe")
| where AdditionalFields contains "PROCESS_VM_WRITE" or AdditionalFields contains "PROCESS_CREATE_THREAD"
| where InitiatingProcessFileName != "MsSense.exe"
```

### 3. Memory Manipulation
**Alert On**: WriteProcessMemory calls targeting security modules
```
DeviceEvents
| where ActionType == "WriteProcessMemoryApiCall"
| where TargetProcessFileName == "MsSense.exe"
| where AdditionalFields contains "CRYPT32.dll"
```

### 4. Proxy Configuration Changes
**Alert On**: netsh winhttp proxy modifications
```
DeviceProcessEvents
| where ProcessCommandLine contains "netsh" and ProcessCommandLine contains "winhttp" and ProcessCommandLine contains "proxy"
| where AccountName != "SYSTEM"
```

### 5. Unauthenticated API Requests
**Alert On**: Network requests to MDE endpoints without proper authentication
```
DeviceNetworkEvents
| where RemoteUrl contains "winatp-gw" and RemoteUrl contains "microsoft.com"
| where InitiatingProcessFileName != "MsSense.exe"
| where InitiatingProcessFileName != "SenseIR.exe"
```

### 6. Command Interception Patterns
**Alert On**: Duplicate or rapid requests to command endpoints
```
DeviceNetworkEvents
| where RemoteUrl contains "/edr/commands/cnc"
| summarize RequestCount = count() by DeviceName, InitiatingProcessFileName, bin(TimeGenerated, 10s)
| where RequestCount > 10
```

### 7. CloudLR Token Abuse
**Alert On**: Unauthorized token requests
```
DeviceNetworkEvents
| where RemoteUrl contains "/senseir/v1/actions/"
| where InitiatingProcessFileName != "SenseIR.exe"
```

### 8. Configuration Exfiltration
**Alert On**: Large data transfers from configuration endpoints
```
DeviceNetworkEvents
| where RemoteUrl contains "/edr/commands/cnc"
| where BytesReceived > 5000000  // 5MB+
| where InitiatingProcessFileName != "MsSense.exe"
```

### 9. Isolation Bypass
**Alert On**: Network activity from "isolated" devices
```
DeviceInfo
| where OnboardingStatus == "Isolated"
| join kind=inner (
    DeviceNetworkEvents
    | where RemoteIPType == "Public"
) on DeviceId
| where TimeGenerated > ago(1h)
```

### 10. Behavioral Correlation
**Alert On**: Multiple suspicious activities in succession
```
let SuspiciousProcesses = DeviceProcessEvents
| where ProcessCommandLine contains "MsSense" or ProcessCommandLine contains "netsh winhttp";
let SuspiciousNetwork = DeviceNetworkEvents
| where RemoteUrl contains "winatp-gw" and InitiatingProcessFileName != "MsSense.exe";
SuspiciousProcesses
| join kind=inner SuspiciousNetwork on DeviceId
| where SuspiciousProcesses.TimeGenerated - SuspiciousNetwork.TimeGenerated < 5m
```

## Expected Results

### Unprotected System (Code 101) - VULNERABLE
- **Phase 2**: MsSense.exe process successfully enumerated
- **Phase 3**: Privileged process handles acquired (PROCESS_VM_WRITE granted)
- **Phase 4**: CRYPT32.dll located, function addresses resolved
- **Phase 5**: Memory patches successfully written to CertVerifyCertificateChainPolicy
- **Phase 7**: Commands successfully intercepted before legitimate agent
- **Phase 8**: CloudLR token generated without authentication
- **Phase 9**: Isolation spoofed, device remains operational
- **Phase 10**: 8MB configuration file exfiltrated
- **Exit Code**: 101 (Unprotected)
- **Risk**: Complete EDR compromise - process injection, memory manipulation, and API bypass all successful

### Protected System (Code 126/127) - SECURE
- **Phase 2**: Process enumeration may succeed (legitimate debugging capability)
- **Phase 3**: Privileged handle acquisition DENIED (Access Denied errors)
- **Phase 5**: Memory writes BLOCKED by security protections
- **Phase 7**: API requests return 401/403 Unauthorized
- **Phase 8**: Token generation fails with authentication errors
- **Phase 10**: Configuration access denied
- **Exit Code**: 126 (Execution Prevented) or 127 (Quarantined)
- **Security**: EDR properly protected against process injection and API exploitation

### Partial Protection (Mixed Results) - NEEDS ATTENTION
- **Scenario 1**: Process protected BUT API vulnerable
  - Phase 3-5: Injection blocked
  - Phase 7-10: API exploitation successful
  - **Risk**: Remote exploitation still possible via API

- **Scenario 2**: API protected BUT process vulnerable
  - Phase 3-5: Injection successful
  - Phase 7-10: API properly authenticated
  - **Risk**: Local attacker can disable EDR via memory patching

- **Exit Code**: Depends on which protection failed
- **Recommendation**: Both process and API protections required for security

## Safety Requirements

### Mandatory Precautions
1. **Isolated Environment**: VM with network isolation or lab network only
2. **VM Snapshot**: Take snapshot before test execution
3. **Administrator Privileges**: Required for process injection operations
4. **MDE Installation**: Test requires MDE to be installed and running
5. **Emergency Access**: Keep emergency_restore.ps1 accessible
6. **Monitoring**: Verify watchdog is operational before aggressive testing

### Recovery Procedures
- **Automatic**: Watchdog auto-restores patches after 30 seconds
- **Deferred**: Cleanup functions run on normal exit or panic
- **Manual**: Execute `emergency_restore.ps1 -Force` as Administrator

### Risk Assessment
- **Memory Patches**: Auto-rollback after 30s, watchdog monitored
- **Process Injection**: No permanent modifications, handles auto-closed
- **API Testing**: Read-only operations, no data modification
- **Overall Risk**: Medium (with safety mechanisms), High (without)
