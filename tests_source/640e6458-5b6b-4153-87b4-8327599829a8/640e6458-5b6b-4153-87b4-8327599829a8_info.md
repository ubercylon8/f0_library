## Executive Summary

This test simulates the **SilentButDeadly** EDR evasion technique, which leverages the Windows Filtering Platform (WFP) to isolate endpoint security products from their cloud infrastructure without terminating processes. Unlike traditional "EDR killing" techniques that stop security processes and trigger immediate alerts, this approach is significantly stealthier by maintaining process execution while severing cloud connectivity.

The technique is particularly dangerous because:
- EDR processes appear operational
- Local monitoring may continue to function
- Cloud-based threat intelligence updates are blocked
- Telemetry uploads fail silently
- Remote management and response capabilities are disabled

## Threat Actor Profile

**Sophistication Level:** Advanced
**Typical Users:**

- Ransomware operators with sophisticated TTPs

**Real-World Usage:** This technique has been observed in advanced intrusion campaigns where threat actors need to operate undetected for extended periods while preventing EDR telemetry from reaching security operations centers (SOCs).

### Stage 2: WFP Filter Application

**Objective:** Block EDR cloud connectivity using legitimate Windows APIs

**Implementation:** Full SilentButDeadly technique port with actual network blocking

**Windows Filtering Platform Architecture:**

The Windows Filtering Platform is a deep packet inspection API that allows applications to inspect and filter network traffic at multiple layers of the network stack. This test implements the complete SilentButDeadly technique that actually creates WFP filters to prevent EDR processes from communicating with cloud infrastructure.

**Key WFP Components:**
1. **Engine** - Core filtering service (fwpuclnt.dll)
2. **Provider** - Entity that adds filters (test creates custom provider)
3. **Sublayer** - Grouping mechanism for related filters
4. **Filters** - Actual blocking rules applied per process (4 per EDR)

**Filter Configuration:**
- **Layers:** 4 layers per EDR process
  - FWPM_LAYER_ALE_AUTH_CONNECT_V4 (IPv4 outbound)
  - FWPM_LAYER_ALE_AUTH_CONNECT_V6 (IPv6 outbound)
  - FWPM_LAYER_ALE_AUTH_RECV_ACCEPT_V4 (IPv4 inbound)
  - FWPM_LAYER_ALE_AUTH_RECV_ACCEPT_V6 (IPv6 inbound)
- **Direction:** Bidirectional (both inbound and outbound)
- **Protocol:** IPv4 and IPv6
- **Action:** Block + Terminating
- **Condition:** Application-specific using AppID blobs

**WFP APIs Used:**
- `FwpmEngineOpen0` - Open WFP engine session
- `FwpmProviderAdd0` - Register filter provider
- `FwpmSublayerAdd0` - Create filter sublayer
- `FwpmGetAppIdFromFileName0` - Convert process path to AppID blob
- `FwpmFilterAdd0` - Add blocking filters (4 per process)
- `FwpmFreeMemory0` - Free AppID blob memory
- `FwpmFilterDeleteById0` - Remove filters (cleanup)
- `FwpmEngineClose0` - Close engine session
- `QueryFullProcessImageNameW` - Get full process paths

**Detection Opportunities:**
- Windows Event Log: Event IDs 5441, 5157, 5152
- WFP provider creation with unusual names
- Filter application targeting security products
- Administrative privileges required (raises suspicion)
- Network connectivity loss for EDR processes

### Stage 3: Service Disruption

**Objective:** Stop and disable EDR services to prevent restart

**Windows APIs Used:**
- `mgr.Connect()` - Connect to Service Control Manager
- `mgr.OpenService()` - Open service handle
- `s.Query()` - Query service status
- `s.Control(svc.Stop)` - Attempt to stop service
- `s.UpdateConfig()` - Modify service startup type

**Service Targeting:**

**Example Service Mappings:**

**Protection Mechanisms Tested:**

2. **Service Stop Prevention** - Can services be stopped?
3. **Service Configuration Lock** - Can startup type be changed?

**Detection Opportunities:**

- Service Control Manager API abuse
- Access denied errors (indicates protection)
- MITRE ATT&CK: T1489 (Service Stop)

## Detection and Response

### Behavioral Indicators

1. **Process Enumeration Pattern**
   - Rapid enumeration of all running processes
   - Focus on security product process names
   - Correlation with subsequent WFP or service operations

2. **WFP Manipulation**
   - Non-standard application creating WFP filters
   - Filter targets include known EDR processes
   - Filter action is "Block" rather than "Permit"
   - Filter weight set to maximum priority

3. **Service Manipulation Cascade**
   - Multiple service stop attempts in sequence
   - Targeting only security-related services
   - Service disable attempts following stop attempts

4. **Network Connectivity Anomaly**
   - EDR process running but no cloud connectivity
   - Telemetry upload failures
   - Heartbeat loss to management console

### Windows Event Logs

**Security Event IDs:**
- **5441** - WFP filter change
- **5157** - WFP connection block
- **5152** - WFP packet drop
- **4656/4658** - Service object handle operations
- **7035/7036** - Service state changes

**Sysmon Event IDs:**
- **Event ID 1** - Process creation (test binary)
- **Event ID 10** - Process access (EDR enumeration)
- **Event ID 22** - DNS query (may be blocked)
- **Event ID 3** - Network connection (should fail for EDR)

### EDR/XDR Detection Strategies

**Recommended Detection Rules:**

1. **WFP Filter Creation Targeting Security Products**
   ```
   Event: WFP filter added (5441)
   Condition: Filter.Target IN [edr_process_list]
   Action: Block, Alert, Terminate process
   ```

2. **Service Manipulation Cascade**
   ```
   Event: Multiple service stop attempts
   Condition: >3 services in <60 seconds AND services IN [security_services]
   Action: Block, Alert, Isolate endpoint
   ```

3. **Process Access Pattern**
   ```
   Event: Process access (Sysmon 10)
   Condition: GrantedAccess = PROCESS_QUERY_INFORMATION AND Target IN [edr_processes]
   Action: Alert, Monitor
   ```

## Exit Code Analysis

### Exit 126 - Execution Prevented (PROTECTED)

**Interpretation:** EDR self-protection mechanisms successfully prevented service manipulation.

**Indicators:**
- Service stop operations returned "Access Denied"
- Service disable operations blocked
- WFP filter creation may have been blocked
- EDR cloud connectivity maintained

**Recommendation:** System is adequately protected against this technique. No immediate action required.

### Exit 101 - Unprotected (VULNERABLE)

**Interpretation:** EDR services were successfully disrupted and isolated from cloud.

**Indicators:**
- One or more EDR services stopped successfully
- Service startup configuration changed to "Disabled"
- WFP filters successfully applied
- EDR isolated from cloud infrastructure

**Recommendation:**
- IMMEDIATE: Restore EDR connectivity and service state
- Review EDR configuration for self-protection features
- Investigate how attack binary executed
- Check for additional malicious activity during EDR isolation window

### Exit 999 - Test Error

**Interpretation:** Test prerequisites not met or execution failed.

**Common Causes:**
- Not running with Administrator privileges

## Response Playbook

### Immediate Actions (0-15 minutes)

1. **Isolate Affected Endpoint**
   - Disconnect from network if EDR isolation confirmed
   - Prevent lateral movement

2. **Restore EDR Functionality**
   ```powershell
   # Restart EDR services (example for Defender)
   Start-Service -Name WinDefend
   Start-Service -Name Sense

   # Verify cloud connectivity
   Test-NetConnection -ComputerName <edr-cloud-endpoint>
   ```

3. **Remove WFP Filters**
   ```powershell
   # List all WFP filters
   netsh wfp show filters

   # Remove suspicious filters (requires filter ID)
   # Or reboot system to clear non-persistent filters
   ```

### Investigation Actions (15-60 minutes)

1. **Timeline Analysis**
   - When did EDR lose cloud connectivity?
   - What processes were running at that time?
   - Were there failed authentication attempts?

2. **Process Tree Analysis**
   - Parent process of attack binary
   - How was it delivered (email, download, lateral movement)?
   - Were other binaries dropped?

3. **Network Analysis**
   - Review firewall logs for blocked connections
   - Check for C2 communications during EDR isolation
   - Analyze DNS queries for suspicious domains

4. **File Analysis**
   - Submit attack binary to sandboxes
   - Calculate file hashes and search threat intel
   - Review code signing certificate

### Remediation Actions (1-4 hours)

1. **Configuration Hardening**
   - Enable EDR tamper protection
   - Implement application whitelisting
   - Restrict WFP API access
   - Monitor Service Control Manager access

2. **Detection Deployment**
   - Deploy WFP filter monitoring rules
   - Alert on service manipulation patterns
   - Monitor EDR cloud connectivity status

3. **Threat Hunting**
   - Search for similar binaries across environment
   - Hunt for WFP filter anomalies historically
   - Review other endpoints for EDR connectivity issues

## Prevention Strategies

### EDR Configuration

1. **Enable Tamper Protection**
   - Microsoft Defender: Enable Tamper Protection in Windows Security
   - CrowdStrike: Enable Self-Protection in policy
   - SentinelOne: Enable Agent Self-Protection

2. **Service Hardening**
   - Set service recovery options to restart automatically
   - Configure service dependencies to prevent isolated stops
   - Use service-specific ACLs to restrict control

3. **Network Resilience**
   - Implement redundant cloud connectivity paths
   - Monitor EDR heartbeat status continuously
   - Alert on telemetry upload failures

### System Hardening

1. **Application Control**
   - Deploy AppLocker or Windows Defender Application Control
   - Whitelist approved applications only
   - Block execution from user-writable directories

2. **Privilege Management**
   - Minimize local administrator accounts
   - Implement Just-In-Time (JIT) admin access
   - Use Privileged Access Workstations (PAWs)

3. **API Monitoring**
   - Monitor WFP API calls from non-standard processes
   - Alert on Service Control Manager access patterns
   - Log process enumeration activities
