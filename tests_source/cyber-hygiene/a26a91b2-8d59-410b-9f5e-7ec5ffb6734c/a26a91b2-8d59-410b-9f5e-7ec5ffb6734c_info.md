## Audience

CISA-credentialed auditors performing IT General Controls validation on Active Directory. This bundle complements the ISACA ITGC Windows Endpoint bundle by covering the 3 ITGCs that require AD-side queries.

## Controls

### ITGC-AM-003 — Dormant Account Detection

**What it does**: Enumerates enabled AD users whose `lastLogonTimestamp` is older than N days (default 90; configurable via `ITGC_DORMANT_DAYS` env var). Excludes service-account OU patterns to avoid false positives.

**Pass criteria**: `dormant_count == 0`

**Manual residual**: Each dormant account must have documented business justification (break-glass, decommissioned-but-retained, compliance-hold). Auditor reviews `evidence.sample_dormant` (first 25 entries) for exception list.

**Mappings**: CISA D5 / COBIT DSS05.04 / CIS 5.3 / MITRE T1078.001

### ITGC-AM-004 — Service Account Permissions Audit

**What it does**: Identifies SPN-holders (i.e. service accounts) that are either (a) members of a privileged group (Domain/Enterprise/Schema Admins, Account/Backup Operators) OR (b) have `PasswordNeverExpires=true`. Either flag is a finding.

**Pass criteria**: `flagged_count == 0`

**Manual residual**: For each flagged account, auditor verifies if a documented exception exists. Recommendation is gMSA migration where feasible.

**Mappings**: CISA D5 / COBIT DSS05.04 / CIS 5.4 / MITRE T1078.002, T1558.003

### ITGC-NS-003 — LAPS Deployment Validation

**What it does**: Two-part check:
1. **Schema check** — does the directory have either `ms-Mcs-AdmPwd` (legacy LAPS) or `msLAPS-Password` (Windows LAPS) attributes registered?
2. **Coverage check** — what fraction of enabled computer objects have a populated LAPS password attribute?

**Pass criteria**: Schema deployed AND `coverage_pct ≥ 80%`

**Manual residual**: Auditor verifies LAPS rotation cadence (default 30d) and read-delegation against approved admin groups.

**Mappings**: CISA D5 / COBIT DSS05.04 / CIS 5.2 / MITRE T1078.003

## Prerequisites

- **Domain-joined Windows host** with RSAT ActiveDirectory PowerShell module
- **Read-only AD account** (Domain Users sufficient for AM-003/AM-004; for LAPS attribute presence check, no special permission needed; for the password value itself — out of scope here — needs LAPS read delegation)
- Network access to a domain controller (TCP 389/636 LDAP, 88 Kerberos, 9389 ADWS)
- PowerShell 5.1+

## Pre-flight

The bundle calls `ADPreFlight()` first, which validates:
1. PowerShell available
2. ActiveDirectory module installed (RSAT)
3. Machine is domain-joined
4. Can query AD (test `Get-ADDomain`)

If any pre-flight check fails, the bundle exits 999 (Test Error) with a clear remediation message — no false-fail evidence is produced.

## Exit codes

- **126** — All 3 controls compliant
- **101** — One or more controls failed (auditor review required)
- **999** — AD pre-flight failed (not domain-joined, missing RSAT, DC unreachable)

## Architecture

Single-binary deployment. The 3 checks share an AD/LDAP session via `RunADCommand` (PowerShell `Import-Module ActiveDirectory` once per check; the AD module pools its own sessions). No multi-binary embed — quarantine resilience isn't relevant here since all checks use the same standard AD attack surface.
