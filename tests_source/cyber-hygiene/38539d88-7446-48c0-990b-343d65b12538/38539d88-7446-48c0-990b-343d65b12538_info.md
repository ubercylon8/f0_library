# Network Protocol Hardening Validator

## Test Information

**Test ID**: 38539d88-7446-48c0-990b-343d65b12538
**Test Name**: Network Protocol Hardening Validator
**Category**: Cyber Hygiene / Configuration Validation
**Severity**: High
**MITRE ATT&CK**: T1557.001, T1040, T1557

## Description

This cyber hygiene test validates that dangerous legacy network broadcast protocols are disabled on Windows endpoints. These protocols - LLMNR, NetBIOS over TCP/IP, and WPAD - are routinely exploited by attackers using tools like Responder to perform man-in-the-middle attacks and harvest credentials on local networks.

The test performs READ-ONLY validation of three critical security configurations:

1. **LLMNR Disabled**: Link-Local Multicast Name Resolution should be disabled via Group Policy to prevent LLMNR poisoning attacks
2. **NetBIOS Disabled**: NetBIOS over TCP/IP should be disabled on ALL network adapters to prevent NBT-NS poisoning
3. **WPAD Mitigated**: Web Proxy Auto-Discovery should be disabled or mitigated to prevent WPAD proxy injection attacks

This test does NOT simulate any attacks - it only validates that defensive configurations are in place.

## Technical Details

### Attack Flow

This is a configuration validation test, not an attack simulation. The "flow" represents the validation sequence:

1. **Phase 0: Initialization**
   - Verify administrator privileges (required for registry access)
   - Initialize Schema v2.0 compliant logging
   - Create output directory c:\F0

2. **Phase 1: LLMNR Check**
   - Open registry key: `HKLM\SOFTWARE\Policies\Microsoft\Windows NT\DNSClient`
   - Read `EnableMulticast` DWORD value
   - Validate: value = 0 means COMPLIANT (LLMNR disabled)
   - Value = 1 or missing means NON-COMPLIANT (LLMNR enabled by default)

3. **Phase 2: NetBIOS Check**
   - Enumerate all subkeys under: `HKLM\SYSTEM\CurrentControlSet\Services\NetBT\Parameters\Interfaces`
   - Filter to `Tcpip_*` interfaces (active network adapters)
   - For each adapter, read `NetbiosOptions` DWORD value
   - Validate: value = 2 on ALL adapters means COMPLIANT
   - Any adapter with value 0 (default) or 1 (enabled) means NON-COMPLIANT
   - Resolve friendly adapter names from network registry

4. **Phase 3: WPAD Check**
   - Method 1: Check `HKLM\...\Internet Settings\WpadOverride` = 1
   - Method 2: Check if `WinHttpAutoProxySvc` service is disabled (Start=4)
   - Method 3: Check if auto-detect flag is cleared in `DefaultConnectionSettings`
   - ANY method being true means COMPLIANT
   - All methods false means NON-COMPLIANT

5. **Phase 4: Compliance Determination**
   - Calculate overall compliance (all 3 checks must pass)
   - Generate summary report with per-adapter NetBIOS details
   - Provide specific remediation guidance for failed checks

### Key Indicators

Security teams monitoring for configuration validation activity may observe:

- Registry reads to `HKLM\SOFTWARE\Policies\Microsoft\Windows NT\DNSClient`
- Registry enumeration of `HKLM\SYSTEM\CurrentControlSet\Services\NetBT\Parameters\Interfaces`
- Registry reads to service configuration keys
- Registry reads to `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Internet Settings`
- Registry reads to `HKCU\Software\Microsoft\Windows\CurrentVersion\Internet Settings\Connections`

## Detection Opportunities

1. **Registry Access Monitoring**
   - Monitor for read access to DNSClient policy key
   - Monitor for enumeration of NetBT Interfaces subkeys
   - Monitor for reads to Internet Settings keys

2. **Reconnaissance Pattern Detection**
   - Sequential reads to multiple security-configuration registry paths
   - Service status checks to WinHttpAutoProxySvc

3. **Process Monitoring**
   - External binary reading multiple registry hives
   - High-privilege process performing configuration enumeration

## Expected Results

### Compliant System (Code 126)

All three protocol hardening configurations are in place:
- LLMNR disabled via GPO (EnableMulticast = 0)
- NetBIOS disabled on ALL network adapters (NetbiosOptions = 2)
- WPAD mitigated via at least one method

The system is protected against Responder-style broadcast poisoning attacks.

### Non-Compliant System (Code 101)

One or more hardening configurations are missing:
- LLMNR may be responding to broadcast queries
- NetBIOS may be enabled on some adapters
- WPAD auto-discovery may be active

System is vulnerable to credential harvesting attacks on the local network.

### Test Error (Code 999)

The test could not complete due to:
- Insufficient privileges (must run as Administrator)
- Timeout exceeded (rare for this test)
- Unexpected system error

## Attack Tool Reference

### Responder

Responder is a popular open-source tool used by penetration testers and attackers to perform LLMNR, NBT-NS, and WPAD poisoning attacks:

```bash
# Example Responder command (for educational purposes)
python Responder.py -I eth0 -wrf
```

When these protocols are enabled, Responder can:
- Capture NTLMv1/v2 hashes from authentication attempts
- Perform SMB/HTTP relay attacks
- Inject malicious WPAD proxy configurations

### Mitigation Effectiveness

| Attack Vector | This Test Validates |
|---------------|---------------------|
| LLMNR Poisoning | LLMNR Disabled check |
| NBT-NS Poisoning | NetBIOS Disabled check |
| WPAD Injection | WPAD Mitigated check |
| Responder attacks | All three checks |

When all three checks pass, the system is protected against the most common broadcast-based credential harvesting attacks.
