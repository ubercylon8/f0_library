# LSASS Protection Validator

**Test Name**: LSASS Protection Validator
**Category**: Cyber Hygiene / Configuration Validation
**Severity**: Critical
**MITRE ATT&CK**: T1003.001, T1003.002, T1550.002

## Description

This cyber hygiene test validates that critical LSASS (Local Security Authority Subsystem Service) protection mechanisms are properly configured on Windows endpoints. LSASS is a high-value target for attackers because it stores credentials in memory, making it the primary target for credential theft tools like Mimikatz.

The test performs READ-ONLY validation of three critical security configurations:

1. **RunAsPPL (Protected Process Light)**: Prevents unauthorized processes from accessing LSASS memory
2. **Virtualization Based Security (VBS)**: Hardware-enforced isolation using hypervisor technology
3. **Credential Guard**: Hardware-based credential isolation that stores credentials in a secure virtual environment

This test does NOT simulate any attacks - it only validates that defensive configurations are in place.

## Technical Details

### Attack Flow

This is a configuration validation test, not an attack simulation. The "flow" represents the validation sequence:

1. **Phase 0: Initialization**
   - Verify administrator privileges (required for registry access)
   - Initialize Schema v2.0 compliant logging

2. **Phase 1: RunAsPPL Check**
   - Open registry key: `HKLM\SYSTEM\CurrentControlSet\Control\Lsa`
   - Read `RunAsPPL` DWORD value
   - Validate: value = 1 (UEFI locked) or 2 (not locked) means COMPLIANT
   - Value = 0 or missing means NON-COMPLIANT

3. **Phase 2: VBS Check**
   - Open registry key: `HKLM\SYSTEM\CurrentControlSet\Control\DeviceGuard`
   - Read `EnableVirtualizationBasedSecurity` DWORD value
   - Validate: value = 1 means COMPLIANT
   - Value = 0 or missing means NON-COMPLIANT

4. **Phase 3: Credential Guard Check**
   - Execute PowerShell WMI query against `Win32_DeviceGuard`
   - Parse `SecurityServicesRunning` array
   - Validate: array contains value 2 means COMPLIANT
   - Value 2 not present means NON-COMPLIANT

5. **Phase 4: Compliance Determination**
   - Calculate overall compliance (all 3 checks must pass)
   - Generate summary report with remediation guidance

### Key Indicators

Security teams monitoring for configuration validation activity may observe:

- Registry reads to `HKLM\SYSTEM\CurrentControlSet\Control\Lsa`
- Registry reads to `HKLM\SYSTEM\CurrentControlSet\Control\DeviceGuard`
- WMI queries to `Win32_DeviceGuard` namespace
- PowerShell execution with `-ExecutionPolicy Bypass`

## Detection Opportunities

1. **Registry Access Monitoring**
   - Monitor for read access to Lsa and DeviceGuard registry keys
   - Could indicate reconnaissance activity

2. **WMI Query Monitoring**
   - Monitor for queries to `Win32_DeviceGuard` class
   - Unusual for standard applications to query this class

3. **PowerShell Logging**
   - Script block logging will capture the WMI query
   - Module logging will show CimInstance cmdlet usage

## Expected Results

### Compliant System (Code 126)

All three protection mechanisms are properly configured:
- RunAsPPL = 1 or 2
- VBS Enabled = 1
- Credential Guard running (service 2 active)

The system is hardened against common credential theft techniques.

### Non-Compliant System (Code 101)

One or more protection mechanisms are missing or misconfigured:
- LSASS memory may be accessible to unauthorized processes
- Credentials may be stored in unprotected memory
- Pass-the-hash attacks may be possible

Remediation is required to improve security posture.

### Test Error (Code 999)

The test could not complete due to:
- Insufficient privileges (must run as Administrator)
- Timeout exceeded (rare for this test)
- Unexpected system error
