**Category**: Cyber Hygiene / Configuration Validation
**Severity**: Critical
**MITRE ATT&CK**: T1059.001, T1059.005, T1055, T1566.001

## Description

This cyber-hygiene test validates that all 8 critical Attack Surface Reduction (ASR) rules are properly configured in Block mode on Windows endpoints. ASR rules are a key defensive capability in Microsoft Defender that can prevent up to 93% of common malware execution patterns when properly configured.

The test is **READ-ONLY** and makes no modifications to the system. It checks ASR configuration through two methods:
1. Windows Registry (Group Policy configuration)
2. PowerShell Get-MpPreference (Defender preferences)

## Technical Details

### Attack Flow

1. **Phase 1: Initialization**
   - Create logging directory at C:\F0
   - Initialize Schema v2.0 compliant logger
   - Resolve organization context

2. **Phase 2: Registry Check**
   - Open HKLM\SOFTWARE\Policies\Microsoft\Windows Defender\Windows Defender Exploit Guard\ASR\Rules
   - Query each of the 8 critical ASR rule GUIDs
   - Record value (0=Disabled, 1=Block, 2=Audit, 6=Warn)

3. **Phase 3: PowerShell Verification**
   - Execute Get-MpPreference to query Defender settings
   - Parse AttackSurfaceReductionRules_Ids and AttackSurfaceReductionRules_Actions
   - Cross-reference with registry results

4. **Phase 4: Analysis**
   - Merge results (prefer registry over PowerShell if both found)
   - Count compliant rules (value = 1/Block)
   - Determine overall compliance status
   - Generate remediation guidance if non-compliant

## Detection Opportunities

1. **Compliance Status Monitoring**
   - Track exit code 126 (compliant) vs 101 (non-compliant) over time
   - Identify endpoints drifting from baseline configuration
   - Alert on configuration changes via periodic validation

2. **Per-Rule Status Tracking**
   - Monitor individual ASR rule status in structured logs
   - Identify which specific rules need remediation
   - Track remediation progress across endpoint fleet

3. **Configuration Drift Detection**
   - Compare results over time to detect configuration changes
   - Identify Group Policy or Intune configuration issues
   - Detect potential policy tampering

## Expected Results

### Compliant System (Code 126)

When all 8 ASR rules are configured in Block mode:
- Exit code: 126 (ExecutionPrevented - indicates system is protected)
- All rules show "BLOCK" status in output
- Summary shows "8/8 compliant"
- Log contains SUCCESS messages for each rule

### Non-Compliant System (Code 101)

When one or more rules are missing, disabled, or in Audit/Warn mode:
- Exit code: 101 (Unprotected - indicates security gap)
- Affected rules show "DISABLED", "AUDIT", "WARN", or "NOT FOUND" status
- Summary indicates number of non-compliant rules
- Remediation PowerShell commands provided in output

### Error Conditions (Code 999)

- Unable to access registry (insufficient permissions)
- PowerShell execution blocked
- Test timeout exceeded

## Behavioral Detection

This test does not require behavioral detection queries as it is a configuration validation test that:
- Performs only read operations
- Does not simulate attacks
- Does not drop files or execute malicious code
- Simply validates security hardening settings

For detecting ASR rule modifications (potential tampering), monitor:
- Windows Event ID 1121/1122 (ASR block/audit events)
- Registry modifications to the ASR\Rules key
- Changes to Defender preferences via PowerShell
