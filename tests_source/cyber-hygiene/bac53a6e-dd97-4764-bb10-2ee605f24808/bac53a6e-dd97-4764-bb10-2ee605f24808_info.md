## Description

This cyber hygiene test validates that CrowdStrike Falcon endpoint protection is properly configured with all critical protection features enabled. The test performs READ-ONLY validation of six critical security configurations that form the foundation of endpoint detection and response:

1. **Falcon Sensor Service**: CSFalconService is running and active
2. **Sensor Operational Status**: Sensor is fully provisioned and communicating with cloud
3. **Prevention Mode**: Sensor is configured to prevent (not just detect) threats
4. **Sensor Version**: Running a current, supported sensor version (7.0+)
5. **Cloud Connectivity**: Sensor can communicate with CrowdStrike cloud for threat intelligence
6. **Tamper Protection**: Sensor is protected from malicious tampering

This test is particularly important for organizations using CrowdStrike Falcon as their primary EDR solution. Proper configuration ensures the sensor can effectively detect and prevent sophisticated attacks including ransomware, fileless malware, and advanced persistent threats.

This test does NOT simulate any attacks - it only validates that defensive configurations are in place.

## Technical Details

### Validation Flow

This is a configuration validation test, not an attack simulation. The validation sequence is:

1. **Phase 0: Initialization**
   - Check for administrator privileges (recommended but not strictly required)
   - Initialize Schema v2.0 compliant logging
   - Verify CrowdStrike Falcon is installed (CSFalconService exists)

2. **Phase 1: Retrieve Falcon Status**
   - Query CSFalconService status
   - Read registry for sensor version, CID, provisioning state
   - Check kernel driver status (CSAgent)
   - Determine operational status
   - Save raw status for debugging

3. **Phase 2: Run Configuration Checks**
   - **Check 1**: Falcon Sensor Service (sc query)
   - **Check 2**: Sensor Operational Status (registry ProvisioningState)
   - **Check 3**: Prevention Mode (service/driver checks)
   - **Check 4**: Sensor Version (registry/file version)
   - **Check 5**: Cloud Connectivity (registry connection state)
   - **Check 6**: Tamper Protection (kernel driver status)

4. **Phase 3: Additional Information**
   - Display sensor version, masked CID, install path
   - Log additional sensor metadata

5. **Phase 4: Compliance Determination**
   - Calculate overall compliance (all 6 checks must pass)
   - Generate summary report with remediation guidance
   - Exit with appropriate code (126=COMPLIANT, 101=NON-COMPLIANT)

## Expected Results

### Compliant System (Code 126)

All six protection mechanisms are properly configured:
- Falcon Sensor Service = Running
- Sensor Operational Status = Provisioned
- Prevention Mode = Enabled
- Sensor Version >= 7.0
- Cloud Connectivity = Connected
- Tamper Protection = Enabled

The system has comprehensive EDR protection and is resilient against attacks that attempt to disable the sensor.

### Non-Compliant System (Code 101)

One or more protection mechanisms are missing or misconfigured:
- Sensor service may not be running
- Sensor may not be properly provisioned with cloud
- Prevention policies may not be applied (detect-only mode)
- Sensor version may be outdated and missing critical protections
- Cloud connectivity issues may prevent threat intelligence updates
- Tamper protection may be compromised

Remediation is required to improve security posture.

### Test Error (Code 999)

The test could not complete due to:
- CrowdStrike Falcon not installed (CSFalconService not found)
- Insufficient privileges for some checks
- Timeout exceeded (rare for this test)
- Unexpected system error

## Why CrowdStrike Falcon Configuration Matters

Modern attackers often follow this attack pattern:
1. Initial access via phishing or vulnerability exploitation
2. **Disable or evade security tools** (EDR, antivirus, logging)
3. Establish persistence
4. Execute payload (ransomware, data theft)

A properly configured CrowdStrike Falcon sensor prevents step 2 by:
- Making it impossible to stop the sensor service (tamper protection)
- Detecting attempts to modify sensor components
- Alerting on reconnaissance of security tools
- Blocking known attack techniques in real-time

Without proper configuration:
- The sensor might be in "detect-only" mode, allowing attacks to proceed
- An outdated sensor might miss recently discovered attack techniques
- Cloud disconnect prevents receipt of new threat intelligence
- Disabled tamper protection allows attackers to stop the sensor
