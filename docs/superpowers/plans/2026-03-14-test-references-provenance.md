# Test References & Provenance Tracking — Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a structured "References & Sources" artifact to F0RT1KA intel-driven and mitre-top10 security tests, tracking the original source document, supporting threat intelligence, and related news — enabling full provenance traceability from test back to threat intel. (Not required for cyber-hygiene tests which derive from compliance frameworks.)

**Architecture:** A new `<uuid>_references.md` file is generated by the `sectest-documentation` agent during Phase 2. Source metadata is captured during Phase 1 (`sectest-source-analysis`), propagated through the context payload, and materialized as a structured markdown artifact. ProjectAchilles renders it as a new "References" sidebar section. Optionally, a `SOURCE_URL:` field in the Go metadata header enables ES enrichment for cross-test source querying.

**Tech Stack:** Markdown (agent skill/config files), TypeScript (ProjectAchilles frontend/backend), Python (ES sync script), Go (metadata header)

The `<uuid>_references.md` artifact was introduced on 2026-03-14. Existing tests

Rather than mass-generating references for all existing tests (which would produce
low-quality provenance data), we take a selective approach:

provenance by researching the original threat intelligence.

### Tier 3: Leave as-is
Tests with low scores, informational severity, or cyber-hygiene category can remain
without references — they're typically based on MITRE ATT&CK documentation rather
than specific threat intelligence.
